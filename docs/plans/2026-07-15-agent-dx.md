## Контекст

Анализ прода (easy-ui.pay-offline.ru) показал, как реально работают агенты пользователя: 78 custom-компонентов yandex-pay, продуктовые сценарии CTYP (63 ревизии) и CPQR (24+ ревизии), цикл «publish компонента → пересохранить evidence-прототип → snap → сравнить глазами», рукотворная 59-экранная галерея дизайн-системы (48 ревизий). При этом visual regression пуст, publish-версии не используются, схемы 78 компонентов агент тянет по одному.

Пять улучшений для агентов (правка YpBox исключена по решению пользователя — отдельная задача):

- **F1** — листинг ассетов + usage (сейчас `GET /api/assets` → 405, дедуп вслепую);
- **F2** — фильтр `?designSystem=` в манифесте каталога + команда драйвера `catalog`;

## F3. Дифф ревизий прототипа

**Новый чистый модуль `src/prototype/revisionDiff.ts`**: `diffPrototypeDocs(from, to, opts?): DocDiff` + **собственный safe-canonicalizer** (null-prototype контейнеры / Map + `Object.hasOwn`; общий `src/capture/canonicalJson.ts` НЕ реиспользовать — он теряет `__proto__`-ключи; общий hash-primitive не менять).

Принципы:
- **Map-диффы — entry-массивы** `{key, ...}` (безопасно для `__proto__`/`constructor`, стабильный порядок — ключи сортированы).
- **Value-union с тремя формами**: `{"value": <json>}` | `{"truncated": {"preview": "<≤120 chars>", "chars": n}}` | `{"missing": true}` (порог 160 символов canonical JSON; `missing` отличает отсутствие optional-поля — `note`, `canvas`, `children`, `visible`, `repeat`, `slot`, `description` — от JSON `null`). Директивы `$state/$template/$cond/$asset`, `repeat`, `slot` — opaque canonical JSON.
- **Бюджет — на весь ответ, включая envelope и все строки**: единый счётчик leaf-changes (default 500) + жёсткий кап `Buffer.byteLength(JSON.stringify(response), "utf8")` ≤ 256 KiB. **Каждая недоверенная строка проходит bounding** — не только значения: `message` (≤500 chars + `truncated`), ключи props/state/stateOverrides, element ids/types, screen ids/names (схема не ограничивает их длину — `schema.ts:14,34,51`) — строки >160 chars усечены с маркером; `screenOrder` — только при изменении и ≤100 записей (иначе `omitted`). Финальная сериализация проверяется по byte-size; при превышении секции целиком заменяются `"omitted": true` (от наименее приоритетных: props-значения → elements → screens), summary считается полностью. `screens.added` — `{id, name, elementCount}` без spec. Тесты: большие messages, гигантский props-ключ/element-type, Unicode/control chars, длинный screenOrder — итоговый byte-size ≤ 256 KiB проверяется буквально.
- **Семантика `identical` (фиксируется в docs и коде)**: `docIdentical = deepEqual(parsedA, parsedB)` — порядок object-ключей игнорируется, порядок массивов значим; `identical = docIdentical && pinsEqual && renderInputsEqual`; оба флага вычисляются **до** truncation/budget; `message`, `createdAt`, `figma` намеренно исключены (это метаданные ревизии, не контент).
- **`renderInputs`**: дифф `builtinCatalogHash`, `componentManifestHash`, `designSystemMetaVersion` (из revision-строк). `screenOrder` — только при изменении порядка.
- Порядок: экраны по `to` (removed — по `from`), пины по id. Пустые секции опускаются. Счётчики — `staticElements*` (repeat не порождает document-ID).

**Схема ответа** `GET /api/prototypes/:id/revisions/:rev/diff?against=<rev>` (default `against = rev-1`):

```jsonc
{
  "prototypeId",
  "from": {"rev","message":V,"createdAt"}, "to": {...},
  "doc":   [ {"key": "name|description|device|designSystem|startScreen", "from": V, "to": V} ],
  "state": { "added": [{key,value:V}], "removed": [key], "changed": [{key,from:V,to:V}] },
  "screens": {
    "added": [{"id","name","elementCount"}], "removed": [{"id","name"}],
    "changed": [{
      "id",
      "meta": [ {"key": "name|note|canvas|root", "from": V, "to": V} ],
      "stateOverrides": {added,removed,changed},          // entry-массивы
      "elements": {
        "added": [{"id","type"}], "removed": [{"id","type"}],
        "changed": [{"id", "type?": {from,to},
          "props": {"added": [{key,value:V}], "removed": [key], "changed": [{key,from:V,to:V}]},
          "children?": {from:V,to:V}, "on": {added:[name],removed:[name],changed:[name]},
          "visible?|repeat?|slot?": {from:V,to:V}}]
      }
    }]
  },
  "screenOrder?": {"from": [...], "to": [...]},           // ≤100 записей, иначе omitted
  "pins": { "components": {added:[{id,version}],removed:[{id,version}],changed:[{id,from,to}]}, "assets": {added,removed} },
  "renderInputs?": [ {"key": "builtinCatalogHash|componentManifestHash|designSystemMetaVersion", "from", "to"} ],
  "summary": {"screensAdded","screensRemoved","screensChanged","staticElementsAdded","staticElementsRemoved","staticElementsChanged","identical","docIdentical","truncated","omittedSections":[...]}
}
```
где `V` = `{"value"} | {"truncated": {"preview","chars"}} | {"missing": true}`.

**Роут** — ветка `tail[2]==="diff"` в `routePrototypes`; любые две разные существующие ревизии. Ошибки: 404 `prototype_not_found`/`revision_not_found`, 400 `invalid_request` (`against===rev`; `rev===1` без `against`). Контракт со строгой схемой и всеми ошибками.

**Безопасность**: сузить prototype-scope allowlist capture-session с prefix `/api/prototypes/` до точных URL capture-shell (exact-matcher в `sessions.ts:30` это поддерживает). Тесты: diff под BasicAuth / share-cookie / capture-token; adversarial-тест capture-протокола не деградирует.

**Тесты**: `server/prototype-diff.test.ts` — 3 ревизии с известными правками → точный JSON; default `against`; `identical`/`docIdentical` (в т.ч. смена designSystemMetaVersion при идентичном doc); ошибки; budget/byte cap/`omitted`; reorder детей и экранов; repeat.key/statePath; slot; вложенный `$cond`; Unicode; `__proto__`/`constructor`-ключи (adversarial: ключ обязан появиться в entry-массиве, не исчезнуть). Юнит `src/prototype/revisionDiff.test.ts` (vitest) — bounding/детерминизм/missing-union/canonicalizer.

**Драйвер `diff <protoId> [revA] [revB]`** (`ranges: [1,3]`, boolean `--json`): 0 арг — head vs head−1 (ошибка при headRev 1); 1 — head vs revA; 2 — revB vs revA. Компактная текстовая сводка.

## F4. Visual baseline/check: серверная модель baseline-set

### Модель (blocker'ы раундов 2–3: атомарность, membership, incarnation, ownership)

**Incarnation (blocker R3-2, ABA)**: миграция v13 добавляет `prototypes.instance_id TEXT NOT NULL` — случайный immutable id, генерируется при create (бэкфилл существующим строкам). Возвращается в meta/draft (`prototypeInstanceId`). `fingerprintSchema` получает optional `prototypeInstanceId` — managed-references включают его (старые references не затронуты). Пересозданный с тем же slug прототип получает новый instance_id → stale PUT и «оживление» старых reference-ID с их историей runs исключены.

**Миграция v13** (кроме instance_id) — таблица `visual_baseline_sets(id TEXT PK, prototype_id TEXT NOT NULL, prototype_instance_id TEXT NOT NULL, generation INTEGER NOT NULL, rev INTEGER NOT NULL, members_json TEXT NOT NULL, created_at TEXT NOT NULL, UNIQUE(prototype_id, generation))`. `members_json` — массив `{screenId, viewport, deviceScaleFactor, theme, referenceId}`, **отсортированный по каноническому surface-key целиком** (не только ключи viewport). FK на prototypes не вводим (как у references), но **hard delete прототипа — инвариант уровня repo**: `PrototypeRepo.delete()` в своей транзакции собирает reference-IDs всех поколений, tombstone'ит их и удаляет sets (прямые callers repo не обойдут cleanup; `server/repos/prototypes.ts` — в ownership W1).

**`PUT /api/visual-baselines/prototypes/:id`** — атомарный batch-commit, **одна SQLite-транзакция `BEGIN IMMEDIATE`** (UNIQUE/lock-конфликты маппятся в 409, не в 500):
- body `{rev, prototypeInstanceId, baseGeneration: number|null, members: [{screenId, viewport, deviceScaleFactor, theme, assetId}]}`;
- CAS: `prototypeInstanceId` обязан совпадать с текущим instance_id прототипа (иначе 409 `instance_conflict`); `baseGeneration` — с текущим максимальным generation этого instance (или null) → 409 `generation_conflict` c `currentGeneration` в details;
- валидация: ревизия существует; **members покрывают все экраны ревизии, ровно одна surface на экран** (недостающий/лишний/дублирующийся экран → 422 `incomplete_baseline`; `minItems: 1`, `maxItems: 64`) — v1 не поддерживает partial sets; каждый asset существует и `image/png`; viewport — целые в границах скриншотов и ≤20 Mpx с учётом dsf (`invalid_viewport`);
- **порядок в транзакции (blocker R3-1)**: сначала tombstone всех references предыдущего поколения, **затем** upsert новых members (upsert того же fingerprint штатно реанимирует строку — повторный baseline той же ревизии работает), затем insert set с `generation+1`;
- ответ `{generation, rev, members: [{screenId, viewport, deviceScaleFactor, theme, referenceId}]}` (полные surface).

**`GET /api/visual-baselines/prototypes/:id`** → последний committed set `{generation, rev, prototypeInstanceId, createdAt, members: [...]}` (та же схема members, что в PUT-ответе); 404 `baseline_not_found` при существующем прототипе без наборов, 404 `prototype_not_found` при отсутствующем прототипе — различаются кодом ошибки.

**Managed-ownership (blocker R3-3)**: generic `PUT /api/visual-references` и `DELETE /api/visual-references/:id` для reference, входящего в последний committed set, → 409 `baseline_managed` (проверка атомарна, в той же транзакции мутации); batch-endpoint использует внутренний привилегированный путь repo. Visual UI (`VisualPage.tsx`) обрабатывает 409 понятным сообщением («управляется baseline-набором»).

Свойства: crash до PUT оставляет только orphan PNG-assets (в docs; кандидаты на будущий GC); crash внутри PUT невозможен частично; конкуренция — CAS 409; удалённые/переименованные экраны и смена canvas — новое поколение заменяет весь membership; **одна активная конфигурация на прототип** (baseline `--theme dark` заменяет light-набор; независимые профили — вне скоупа v1, явно зафиксировано в docs/SKILL.md).

**Серверные тесты транзакции** — новый `server/visual-baselines.test.ts`: rollback при искусственном сбое внутри транзакции; повторный baseline той же rev (полное и частичное пересечение surfaces — reference реанимируется, set указывает на активные строки); stale generation → 409; stale instanceId (delete/recreate) → 409; пустой/частичный/дублирующий members → 422; invalid viewport (включая 20 Mpx×dsf) → 422; delete-cleanup через repo; generic PUT/DELETE на managed reference → 409; порядок members нормализован при любом порядке на входе; runs-история сохраняется у tombstoned references; auth: GET/PUT baseline доступны под BasicAuth и недоступны под share-cookie/capture-token.

### Candidate-override и candidateMeta

- `VisualService.check(referenceId, {threshold?, rev?, version?})`: `rev` — только prototype-screen, `version` — только component; неверная пара → 422 `invalid_candidate_target`. **Любой check по fingerprint с `propsHash`/`stateHash` → 422 `invalid_candidate_target`** (не только override: рецепта воспроизведения нет и без него кандидат снимается с `{}`; снятие ограничения — вместе с capture-recipe, вне скоупа v1).
- `enqueue`: `rev: candidate.rev ?? fp.refRevision` / `version: candidate.version ?? fp.refVersion`.
- **Канал frozen snapshot — внутренний метод**: `ScreenshotService.enqueueWithExpected(...)` возвращает `{jobId, expected}` только VisualService'у; **публичные routes (`server/routes/screenshots.ts`) не меняются и продолжают отдавать строго `{jobId}`** — HTTP-тест на отсутствие `expected` в ответе. VisualService сохраняет `requestedTarget` + expected сразу после enqueue — метаданные полны и на timeout/error.
- **Browser-ошибки — терминальный `error`**: `consoleErrors`/`pageErrors` кандидата → run status `error` c bounded diagnostics (не «pass по пикселям»), симметрично abort-политике baseline.
- candidateMeta — **discriminated union по `kind` (prototype|component) и `outcome` (captured|capture_failed)**: `{requestedTarget, resolvedTarget, expected, browser: {...}|null}` — `browser: null` для pre-browser ошибок (timeout до навигации и т.п.); **все существующие top-level поля сохраняются** (`rev/version/pins/bundleHash/rendererBuild/browserVersion` — как aliases, публичные поля не удаляются).
- Контракт check += `{rev?, version?}`; полная error-матрица: 404 (reference/prototype/screen/revision/version not found), 422 (`invalid_candidate_target`, `invalid_threshold`, **`invalid_viewport`** — fingerprint допускает размеры, которые screenshot отвергает: любые положительные vs диапазон+20 Mpx), 429 `queue_full`, 501.
- Тесты (`server/visual.test.ts`, fake ScreenshotService): override rev/version; без override; requested/resolved на success и на error/timeout (browser:null); hash-bearing fingerprint → 422; browser errors → `error`; hash-несовместимый viewport → `invalid_viewport`; все legacy-поля candidateMeta на месте; HTTP-ответ screenshot-роутов без `expected`.

### Драйвер

**`baseline <protoId> [outDir]`** (`ranges: [1,2]`, флаги `--viewport WxH`, `--theme light|dark`, `--dsf 1|2|3`):
1. Один `GET /prototypes/:id/draft` — snapshot: `draft.rev`, `prototypeInstanceId`, device, экраны, canvas; `GET /visual-baselines/...` → `baseGeneration` (`baseline_not_found` → null).
2. Viewport per screen: `--viewport ?? screen.canvas (Math.round до целых, клэмп в границы 64–2000 × 64–4000) ?? canonical device (mobile 390×844, tablet 834×1112) ?? desktop 1280×800`; до enqueue драйвер проверяет полный инвариант **width×height×dsf² ≤ 20 Mpx**. Desktop-значение — **новая политика** (canonical desktop = auto), константа драйвера, фиксируется в SKILL.md/docs.
3. Последовательно по экранам: POST screenshot `{rev: draft.rev, ...}` → общий `pollJob(path, {deadlineMs: 120_000})`; `consoleErrors`/`pageErrors` → abort всего baseline (записей нет, орфанные PNG остаются — задокументировано).
4. Все экраны успешны → **один** `PUT /api/visual-baselines/prototypes/:id` `{rev, prototypeInstanceId, baseGeneration, members}` (по одному member на каждый экран ревизии); 409 (`generation_conflict`/`instance_conflict`) → перечитать состояние, сообщить о гонке (exit 1, без авто-retry). Печать `screenId -> vref_<id>`; PNG в outDir при задании.

**`check <protoId>`** (`ranges: [1,1]`, флаги `--threshold N`, `--json`):
1. `GET /visual-baselines/prototypes/:id` (404 → «run baseline first», exit 1); `GET /draft` → `draft.rev`.
2. Последовательно по members: `POST /visual-references/:referenceId/check {rev: draft.rev, threshold?}` → poll `/visual-runs/:runId`.
3. Таблица `screenId status diffPercent refRev->candRev diffUrl`; exit non-zero при fail/error/reference_missing/timeout.

**Тестируемость (CI, без chromium)**: планировщик baseline/check выносится в чистые функции, экспортируемые из `driver.mjs` (исполнение CLI — под guard `process.argv[1]`); `server/driver-cli.test.ts` импортирует их и покрывает с mocked HTTP: viewport-каскад (canvas-округление, порядок ключей viewport-объекта не влияет), группировка members, 409-гонка, canvas change / removed screen (membership заменяется), hash-bearing reference, abort при browser errors, повторный запуск после сбоя.
