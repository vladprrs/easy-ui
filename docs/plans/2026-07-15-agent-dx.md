# План v2: Agent-DX улучшения easy-ui (2026-07-15)

Раунд 1 ревью (Codex gpt-5.6-sol, config max): 2 blocker / 16 major / 3 minor. Триаж — в конце документа; принято всё, кроме переделки OpenAPI-генератора (частично, см. триаж). Изменения v2 относительно v1 вплетены в секции.

## Контекст

Анализ прода (easy-ui.pay-offline.ru) показал, как реально работают агенты пользователя: 78 custom-компонентов yandex-pay, продуктовые сценарии CTYP (63 ревизии) и CPQR (24+ ревизии), цикл «publish компонента → пересохранить evidence-прототип → snap → сравнить глазами», рукотворная 59-экранная галерея дизайн-системы (48 ревизий). При этом visual regression пуст, publish-версии не используются, схемы 78 компонентов агент тянет по одному.

Пять улучшений для агентов (правка YpBox исключена по решению пользователя — отдельная задача):

- **F1** — листинг ассетов + usage (сейчас `GET /api/assets` → 405, дедуп вслепую);
- **F2** — фильтр `?designSystem=` в манифесте каталога + команда драйвера `catalog`;
- **F3** — дифф двух ревизий прототипа (endpoint + команда `diff`);
- **F4** — команды драйвера `baseline`/`check` + серверные фиксы: candidate-rev override (сейчас check снимает кандидата на той же ревизии, что эталон) и модель baseline-набора;
- **F5** — named examples у компонентов: несколько именованных наборов props → манифест, capture, скриншот-API, чипы вариантов в Library. Заменяет ручную галерею.

## F3. Дифф ревизий прототипа

**Новый чистый модуль `src/prototype/revisionDiff.ts`** (framework-free): `diffPrototypeDocs(from, to, opts?): DocDiff`.

Принципы (по ревью):
- **Все map-диффы — entry-массивы** `{key, value}` / `{key, from, to}` (не `{k:v}`-объекты: безопасно для `__proto__`/`constructor`-ключей, стабильный порядок — ключи сортированы).
- **Значения — недвусмысленный union**: `{"value": <json>}` либо `{"truncated": {"preview": "<≤120 chars>", "chars": n}}` (порог 160 символов canonical JSON). Директивы `$state/$template/$cond/$asset`, `repeat`, `slot` сравниваются как opaque canonical JSON.
- **Глобальный бюджет**: единый счётчик leaf-changes (default 500) на весь ответ (props-ключи, state-ключи, элементы, экраны, пины) + жёсткий кап сериализованного ответа 256 KiB. По исчерпании — значения не эмитятся, секции получают `"omitted": true`, summary продолжает считаться полностью. `screens.added` — только `{id, name, elementCount}`, без полного spec.
- **`renderInputs`**: дифф `builtinCatalogHash`, `componentManifestHash`, `designSystemMetaVersion` (из revision-строк, не из doc). `summary.identical` = doc-JSON идентичен **и** пины **и** renderInputs не изменились; отдельный флаг `docIdentical`.
- `screenOrder: {"from": [...], "to": [...]}` — только если порядок массива screens изменился.
- Порядок: экраны по `to` (removed — по `from`), пины по id. Пустые секции опускаются, `summary` — всегда. В summary счётчики называются `staticElements*` (repeat не порождает новых document-ID).

**Схема ответа** `GET /api/prototypes/:id/revisions/:rev/diff?against=<rev>` (default `against = rev-1`):

```jsonc
{
  "prototypeId",
  "from": {"rev","message","createdAt"}, "to": {...},
  "doc":   [ {"key": "name|description|device|designSystem|startScreen", "from": V, "to": V} ],
  "state": { "added": [{key,value:V}], "removed": [key], "changed": [{key,from:V,to:V}] },
  "screens": {
    "added": [{"id","name","elementCount"}], "removed": [{"id","name"}],
    "changed": [{
      "id",
      "meta": [ {"key": "name|note|canvas|root", "from": V, "to": V} ],
      "stateOverrides": {added,removed,changed},          // entry-массивы
      "elements": {
        "added": [{"id","type"}], "removed": [{"id","type"}],
        "changed": [{"id", "type?": {from,to},
          "props": {"added": [{key,value:V}], "removed": [key], "changed": [{key,from:V,to:V}]},
          "children?": {from,to}, "on": {added:[name],removed:[name],changed:[name]},
          "visible?|repeat?|slot?": {from:V,to:V}}]
      }
    }]
  },
  "screenOrder?": {"from": [...], "to": [...]},
  "pins": { "components": {added:[{id,version}],removed:[{id,version}],changed:[{id,from,to}]}, "assets": {added,removed} },
  "renderInputs?": [ {"key": "builtinCatalogHash|componentManifestHash|designSystemMetaVersion", "from", "to"} ],
  "summary": {"screensAdded","screensRemoved","screensChanged","staticElementsAdded","staticElementsRemoved","staticElementsChanged","identical","docIdentical","truncated","omittedSections":[...]}
}
```
где `V` = `{"value": ...} | {"truncated": {"preview","chars"}}`.

**Роут** — ветка `tail[2]==="diff"` в `routePrototypes`; любые две разные существующие ревизии. Ошибки: 404 `prototype_not_found`/`revision_not_found` (обе стороны), 400 `invalid_request` при `against===rev` и при `rev===1` без `against`. Контракт `prototypeRevisionDiffContract` со строгой схемой и всеми ошибками.

**Безопасность**: сузить prototype-scope allowlist capture-session с prefix `/api/prototypes/` до точных URL, нужных capture-shell (revision/version endpoint конкретной ревизии) — diff и прочие GET по capture-токену недоступны. Тесты: новый GET под BasicAuth / share-cookie / capture-token; adversarial-тест capture-протокола не деградирует.

**Тесты**: `server/prototype-diff.test.ts` — 3 ревизии с известными правками (prop, add/remove элемента, add экрана, state-ключ, doc.name, бамп пина после republish, ассет) → точный JSON; default `against`; `identical`/`docIdentical` (в т.ч. смена designSystemMetaVersion при идентичном doc); ошибки; truncation и глобальный budget/byte cap (`omitted`); **reorder детей и экранов; repeat.key/statePath; slot; вложенный `$cond`; Unicode; ключи `__proto__`/`constructor`**. Юнит `src/prototype/revisionDiff.test.ts` (vitest) — bounding/детерминизм/entry-порядок.

**Драйвер `diff <protoId> [revA] [revB]`** (`ranges: [1,3]`, флаг `--json` boolean): 0 арг — head vs head−1 (ошибка при headRev 1); 1 — head vs revA; 2 — revB vs revA. Компактная текстовая сводка; `--json` — сырой ответ.

## F4. Visual baseline/check в evidence-цикле

**Сервер — candidate-override**:
- `VisualService.check(referenceId, {threshold?, rev?, version?})`: `rev` только для scope `prototype-screen`, `version` — для `component`; неверная пара → 422 `invalid_candidate_target`. **Override запрещён для fingerprints с `propsHash`/`stateHash`** (422 `invalid_candidate_target`): рецепта воспроизведения props/state в reference нет, кандидат снялся бы с другими props (`screenshot/service.ts:138` — `{}` без props). Зафиксировать в docs; снятие ограничения — вместе с capture-recipe в reference (вне скоупа v1).
- `enqueue`: `rev: candidate.rev ?? fp.refRevision` / `version: candidate.version ?? fp.refVersion`.
- **candidateMeta на всех исходах**: в run пишутся `requestedTarget` (что просили), `resolvedTarget` (что фактически снялось) и immutable expected-снапшот capture-протокола (builtinCatalogHash, componentManifestHash, dsMetaVersion, propsHash — что доступно), отдельно browser/renderer metadata; на timeout/error — `requestedTarget` сохраняется тоже (сейчас пишется только `{error}`, `service.ts:133`). На success фактический target — из frozen снапшота скриншот-джоба.
- `server/routes/visual.ts` `checkReference`: `rev`/`version` из body; контракт += `{rev?, version?}`, все ошибки задекларированы (404 reference, 422 target, 429 queue_full).
- Тесты (`server/visual.test.ts`, fake ScreenshotService): `{rev:9}` → enqueue rev 9; без override — refRevision; candidateMeta requested/resolved на success и на error; `{version}` на prototype-scope → 422; override на fingerprint с propsHash → 422.

**Модель baseline-набора (blocker №1)** — решается в драйвере, сервер не меняем:
- Surface = `(prototypeId, screenId, viewport, deviceScaleFactor, theme)`.
- `baseline`: после успешной съёмки **всех** экранов — для каждого surface tombstone'ит (DELETE) существующие references с иным `refRevision`, затем PUT новых. До полной успешной съёмки ничего не пишется (атомарность: сначала все капчи + проверка ошибок, потом записи).
- `check`: группирует references по surface, берёт **только** reference с максимальным `refRevision` в группе (защита от осиротевших старых), остальные показывает как `stale (superseded)` без проверки.
- Тест-сценарий (runtime-верификация): baseline rev N → правка → check fail → baseline rev N+1 → check проверяет только новый набор → pass.

**Драйвер `baseline <protoId> [outDir]`** (`ranges: [1,2]`, флаги `--viewport WxH`, `--theme light|dark`, `--dsf 1|2|3`):
1. **Один** `GET /prototypes/:id/draft` — snapshot: `draft.rev` (никакого отдельного getMeta → нет TOCTOU), device, экраны, `screen.canvas`.
2. Viewport per screen: `--viewport ?? screen.canvas ?? canonical device (mobile 390×844, tablet 834×1112) ?? desktop-фолбэк 1280×800` (согласовано с Visual UI, `VisualPage.tsx:243`).
3. Последовательно по экранам: POST screenshot `{rev: draft.rev, viewport, dsf, theme}` → poll через **общий helper** `pollJob(path, deadlineMs)` (дедлайн 120s, согласован с серверными 60s job + 90s check); `consoleErrors`/`pageErrors` в результате → **abort всего baseline без записи** (non-zero exit).
4. Все экраны успешны → tombstone superseded + `PUT /visual-references` `{fingerprint: {scope:"prototype-screen", prototypeId, screenId, refRevision: draft.rev, viewport, deviceScaleFactor, theme}, assetId, note: "baseline rev N"}`; PNG в outDir, если задан. Печать `screenId -> vref_<id>`.

**Драйвер `check <protoId>`** (`ranges: [1,1]`, флаги `--threshold N`, `--json`):
1. `GET /visual-references?scope=prototype-screen&prototypeId=` (пусто → «run baseline first», exit 1); `GET /draft` → `draft.rev`; группировка по surface, максимальный refRevision.
2. Последовательно: `POST /visual-references/:id/check {rev: draft.rev, threshold?}` → poll `/visual-runs/:runId` тем же helper'ом.
3. Таблица `screenId status diffPercent refRev->candRev diffUrl` (+ строки `stale`); exit non-zero при fail/error/reference_missing/timeout.

## F5. Named examples компонентов
