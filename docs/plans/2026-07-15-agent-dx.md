# План: Agent-DX улучшения easy-ui (2026-07-15)

## Контекст

Анализ прода (easy-ui.pay-offline.ru) показал, как реально работают агенты пользователя: 78 custom-компонентов yandex-pay, продуктовые сценарии CTYP (63 ревизии) и CPQR (24+ ревизии), цикл «publish компонента → пересохранить evidence-прототип → snap → сравнить глазами», рукотворная 59-экранная галерея дизайн-системы (48 ревизий). При этом visual regression пуст, publish-версии не используются, схемы 78 компонентов агент тянет по одному.

Пять улучшений для агентов (правка YpBox исключена по решению пользователя — отдельная задача):

- **F1** — листинг ассетов + usage (сейчас `GET /api/assets` → 405, дедуп вслепую);
- **F2** — фильтр `?designSystem=` в манифесте каталога + команда драйвера `catalog`;
- **F3** — дифф двух ревизий прототипа (endpoint + команда `diff`) — при 63 ревизиях «что изменилось» не видно;
- **F4** — команды драйвера `baseline`/`check` (visual regression в evidence-цикл) + серверный фикс: сейчас check снимает кандидата на той же ревизии, что эталон (сравнение с самим собой);
- **F5** — named examples у компонентов: несколько именованных наборов props → манифест, capture, скриншот-API, чипы вариантов в Library. Заменяет ручную галерею.

После одобрения: сохранить в `docs/plans/2026-07-15-agent-dx.md`, закоммитить, адверсариальное ревью Codex gpt-5.6-sol (config max), триаж, затем исполнение волнами Codex `--write --effort medium`.

## F1. Листинг ассетов + usage

`server/repos/assets.ts`:
- `list({limit, before?})` — keyset-пагинация `(created_at, id) <` , `ORDER BY created_at DESC, id DESC`;
- `usage(id)` — три reverse-запроса: prototypes (`GROUP BY prototype_id`: lastRev, revCount, name, head_rev), components (component_id, version, name), visual references (id по `asset_id`).

`server/routes/assets.ts`:
- `GET /api/assets?limit=&cursor=` (limit default 50, max 200; cursor opaque `"<createdAt>~<id>"`; malformed → 400). Ответ: `{assets: [{id, sha256, mime, size, width?, height?, originalName, createdAt, url, usage: {prototypes, components, visualReferences}}], nextCursor}` — счётчики скалярными COUNT-подзапросами в одном SQL (без N+1).
- `GET /api/assets/:id/usage` → `{asset: {...}, prototypes: [{id, name, revCount, lastRev, pinnedAtHead}], components: [{id, name, versions: [...]}], visualReferences: ["vref_..."]}`; 404 `asset_not_found`.

Контракты `listAssetsContract` + `assetUsageContract` (`validated: true`), regen OpenAPI. Тесты в `server/assets.test.ts`: пустой список; пагинация 3×limit 2 (стабильный порядок, cursor round-trip); usage после `$asset`-пина прототипа и component-publish со ссылкой; usage после PUT visual-reference; 404. Индексы на `*(asset_id)` — не добавляем (масштаб мал), зафиксировать в docs как осознанное.

Драйвер: `get assets` заработает через существующий `get`; спец-кейс `get assets <id>` → `/assets/<id>/usage` (сырые байты в stdout не лить).

## F3. Дифф ревизий прототипа

**Новый чистый модуль `src/prototype/revisionDiff.ts`** (framework-free, сервер уже импортирует `src/prototype/*`):
`diffPrototypeDocs(from, to, opts?: {maxValueChars?, maxChanges?}): DocDiff`.
- Матчинг экранов и элементов по id (алгоритм по образцу `docDiff.ts`, deepEqual).
- Bounding значений: JSON ≤160 символов — как есть; иначе `{$truncated: true, preview: <120 chars>, chars}`. Глобальный кап 400 element-level записей → `summary.truncated`.
- Детерминизм: ключи сортированы, порядок экранов — по `to` (removed — по `from`), пины по id. Пустые секции опускаются, `summary` — всегда.

**Схема ответа** `GET /api/prototypes/:id/revisions/:rev/diff?against=<rev>` (default `against = rev-1`):

```jsonc
{
  "prototypeId", 
  "from": {"rev", "message", "createdAt"}, "to": {...},
  "doc":   { "name|description|device|designSystem|startScreen": {"from","to"} },  // только изменённые
  "state": { "added": {k: v}, "removed": [k], "changed": {k: {"from","to"}} },
  "screens": {
    "added": [{"id","name","elements"}], "removed": [{"id","name"}],
    "changed": [{
      "id",
      "meta": { "name|note|canvas|root": {"from","to"}, "stateOverrides": {added,removed,changed} },
      "elements": {
        "added": [{"id","type"}], "removed": [{"id","type"}],
        "changed": [{"id", "type?": {"from","to"},
          "props": {"added": {k:v}, "removed": [k], "changed": {k: {"from","to"}}},
          "children?": {"from","to"}, "on": {added,removed,changed},
          "visible?|repeat?|slot?": {"from","to"}}]
      }
    }]
  },
  "pins": { "components": {added,removed,changed:[{id,from,to}]}, "assets": {added,removed} },
  "summary": {"screensAdded","screensRemoved","screensChanged","elementsAdded","elementsRemoved","elementsChanged","identical","truncated"}
}
```

**Роут** — ветка `tail[2]==="diff"` в `routePrototypes`; `from = against` (старшая сторона не требуется — любые две разные существующие ревизии). Ошибки: 404 `prototype_not_found`/`revision_not_found` (обе стороны), 400 `invalid_request` при `against===rev` и при `rev===1` без `against`. Контракт `prototypeRevisionDiffContract`, regen OpenAPI.

**Тесты**: `server/prototype-diff.test.ts` — 3 ревизии с известными правками (prop-изменение, add/remove элемента, add экрана, state-ключ, doc.name, бамп пина после republish компонента, добавленный ассет) → точный JSON; default `against`; идентичные ревизии → `identical`; ошибки; truncation (>160 chars). Юнит `src/prototype/revisionDiff.test.ts` (vitest) — bounding/детерминизм.

**Драйвер `diff <protoId> [revA] [revB]`** (`ranges: [1,3]`): 0 арг — head vs head−1 (ошибка при headRev 1); 1 — head vs revA; 2 — revB vs revA. Печать компактной сводки (`~screen/element props.text: "A" -> "B"`), `--json` — сырой ответ.

## F4. Visual baseline/check в evidence-цикле

**Сервер (prerequisite)** — candidate-override:
- `VisualService.check(referenceId, {threshold?, rev?, version?})`: `rev` только для scope `prototype-screen`, `version` — только для `component`; неверная пара → 422 `invalid_candidate_target`.
- `enqueue`: `rev: candidate.rev ?? fp.refRevision` / `version: candidate.version ?? fp.refVersion`.
- `candidateMeta`: писать **фактически снятую** ревизию/версию (пробросить через `drive`), не `fp.refRevision`. Потребители (`repo.runReport`, `src/visual/api.ts`) — loose, безопасно.
- `server/routes/visual.ts` `checkReference`: `rev`/`version` из body; контракт `checkVisualReferenceContract.requestSchema` += `{rev?, version?}` + 422. Regen OpenAPI.
- Тесты (`server/visual.test.ts`, fake ScreenshotService): `{rev:9}` → enqueue с rev 9; без override — `fp.refRevision`; `candidateMeta.rev===9`; `{version}` на prototype-scope → 422.

**Драйвер `baseline <protoId> [outDir]`** (`ranges: [1,2]`):
1. `getMeta` → headRev; `GET /prototypes/:id/draft` → device, экраны.
2. Viewport: mobile 390×844, tablet 834×1112, **desktop → 1280×800 (фолбэк, зафиксировать в SKILL.md)**; флаги `--viewport WxH`, `--theme light|dark` (default light), `--dsf 1|2|3` (default 1).
3. Последовательно по экранам (очередь 5/concurrency 1): POST screenshot `{rev: headRev, viewport, dsf, theme}` → poll job (цикл как в `snap`, ~60×2s) → `PUT /visual-references` `{fingerprint: {scope: "prototype-screen", prototypeId, screenId, refRevision: headRev, viewport, deviceScaleFactor, theme}, assetId, note: "baseline rev N"}`; PNG в outDir, если задан.
4. Печать `screenId -> vref_<id>`; console/page errors → non-zero exit (как snap).

**Драйвер `check <protoId>`** (`ranges: [1,1]`, флаги `--threshold N`, `--json`):
1. `GET /visual-references?scope=prototype-screen&prototypeId=` (пусто → «run baseline first», exit 1); `getMeta` → headRev.
2. Последовательно: `POST /visual-references/:id/check {rev: headRev, threshold?}` → poll `/visual-runs/:runId` (~60×2s; сервер CHECK_DEADLINE 90s).
3. Таблица `screenId status diffPercent refRev->candRev diffUrl`; exit non-zero при fail/error/reference_missing/timeout.

**parseArgs**: заменить одиночную проверку `--design-system` на per-command allowlist флагов (`component|component-move: [--design-system]`, `baseline: [--viewport,--theme,--dsf]`, `check: [--threshold,--json]`, `diff: [--json]`).

## F5. Named examples компонентов

**Пайплайн**:
- `server/components/types.ts`: `CustomComponentDefinition.examples?: Record<name, Props>`; `DefinitionMeta.examples?: Record<string, Record<string, unknown>>`.
- `server/components/extract-subprocess.ts`: metadata strictObject += `examples` optional; валидация: имя — slug `^[a-z0-9]+(?:-[a-z0-9]+)*$`, длина 1–32, ≤8 записей, каждый props-набор — `metadata.props.parse` (ошибка именует example) → 422 `validation_failed`; smoke-render каждого именованного примера (warning при падении); ключи сортируются при эмите.
- `server/components/pipeline.ts` `definitionMeta()`: passthrough `examples`.
- Персист/манифест/version-DTO — **без правок** (спред `definition_meta` проверен).

**Capture** (`src/capture/CaptureComponent.tsx`): `?example=<name>` подразумевает example-режим (`?props=example` остаётся дефолтным вариантом); порядок резолва props: bootstrap → `examples[name]` → `example` → `{}`; неизвестное имя → существующий error-путь (`data-capture-error`).

**Library**: `src/api/client.ts` — `examples?` в `CatalogComponent` и `ComponentVersion`; `LibraryPage.tsx` (`ComponentMetadata`) — ряд чипов вариантов (`default` при `example` + сортированные ключи `examples`), выбранный чип → URL превью; фолбэк `noExampleProps` без обоих. Строки — `src/app/strings/library.ts`.

**Скриншот-API**: `POST /components/:id/versions/:v/screenshot` += `exampleName?` (взаимоисключим с `props` → 400); `enqueueComponent` резолвит `dto.examples[name]` (нет → 422 `unknown_example`) → дальше существующий validate/propsHash-путь. Контракт += поле и ошибка, regen OpenAPI.

**Схема definition** (`server/routes/meta.ts` `buildComponentDefinitionSchema`): добавить `examples` (propertyNames pattern, maxProperties 8) — иначе `additionalProperties:false` запретит поле. Заодно НЕ трогаем pre-existing drift (`interactive` и пр. отсутствуют) — отметить как известное.

**Тесты** (`server/components.test.ts`): publish с валидными examples → в version-DTO и манифесте; невалидное имя → 422; >8 → 422; example, не проходящий props-схему → 422 с именем; screenshot с `exampleName`; unknown → 422.

## Риски

- **candidateMeta.rev меняет семантику** (честная запись снятой ревизии) — потребителей, полагающихся на `rev===refRevision`, в репо нет; отметить в docs.
- **Размер диффа** на патологических доках — кап 400 записей + `$truncated`, константы в `revisionDiff.ts`.
- **404-vs-422** для unknown design system в F2 — осознанная несогласованность, в docs.
- **Очередь скриншотов 5/1** — драйвер строго последователен; на медленном проде baseline 5-экранного прототипа ≈ минуты, это норм.
- Прод-данные не мигрируются (ни одной DB-миграции в плане) — деплой безопасен, бэкап по обычной процедуре всё равно снять.
