# Витрина custom-компонентов в Library: страница компонента с живым превью, контролами и доками

Дата: 2026-07-16 · Статус: draft (на адверсариальное ревью)

## 1. Контекст и цель

Фокус продукта — пользовательские дизайн-системы и компоненты (создаваемые через `POST /api/design-systems` / `POST /api/components`, хранимые в SQLite, компилируемые сервером в браузерные ESM-бандлы). Сейчас у builtin-систем (shadcn/wireframe) есть полноценная витрина — Storybook с Controls и Docs, а custom-компоненты показываются в Library только meta-карточкой с capture-iframe (`src/library/LibraryPage.tsx`, `ComponentMetadata`). Асимметрия принципиальна: Storybook — build-time артефакт и не может показывать runtime-данные из БД.

**Цель:** сделать custom-компоненты первосортными гражданами продукта — страница компонента в Library с:
1. живым превью (тот же рантайм, что у плеера);
2. панелью контролов, сгенерированной из схемы props (аналог Storybook Controls);
3. докстраницей из `definition_meta` (props-таблица, события, слоты, примеры) и вкладкой с исходником (аналог Storybook Docs).

**Не цель (out of scope):** паритетная миграция builtin-компонентов на эту витрину (отдельная задача после MVP), markdown-доки в publish-контракте, изменение publish-пайплайна и `definition_meta`, серверные изменения вообще (весь материал уже отдают существующие API).

Рассмотренные и отклонённые альтернативы:
- **Storybook composition (refs):** даёт дерево+рендер, но Controls/Docs для ref-историй не работают (ограничение модели composition) — а они прямое требование.
- **Отдельный storybook-проект с пересборкой по кнопке:** настоящие Controls/Docs, но минутная задержка publish→витрина ломает основной цикл (роадмап: AI-генерация компонентов), второй рантайм требует синхронизации с плеером (ABI, `wrapCustomComponent`, темы), сборка от пользовательских данных хрупка (один битый компонент валит витрину всех), плюс инфраструктурный хвост (репо, секреты, деплой; сборка на прод-сервере запрещена).

## 2. Ключевые решения

### D1. Превью рендерится инлайн, без iframe и postMessage-канала

Прецедент уже в кодовой базе: плеер (`/p/:id`) и редактор (`/p/:id/edit`) рендерят custom-компоненты инлайн внутри Layout — `loadCustomComponents` (динамический import бандла, `src/customComponents/loader.ts`) → `CaptureSurface`/`Renderer` + `ThemeStyle` + `SurfaceSpacingScope`. Страница компонента делает то же самое, что `CaptureComponent` (`src/capture/CaptureComponent.tsx:65-97`), но в обычном роуте. Живые контролы при этом — просто React-стейт `props` → ре-рендер, никакого межфреймового протокола.

Отклонено: capture-iframe + postMessage-канал live-props. Причины: (а) в capture нет входящего канала (props фиксируются при загрузке, effect с `[]`-deps), пришлось бы проектировать и верифицировать новый протокол с валидацией origin/props — существенный кусок работы; (б) изоляция iframe не нужна — совместимость темы DS с app-шеллом уже доказана плеером/редактором (`ThemeStyle` с restore-on-unmount, `SurfaceSpacingScope` скоупит `space.*`); (в) инлайн проще, быстрее и без задержки перезагрузки iframe на каждое изменение props.

Capture-shell не трогаем — он остаётся контрактом скриншот-воркера и visual regression.

### D2. Контролы — из живой zod-схемы через вынесенное ядро PropsForm

В редакторе уже есть schema-driven форма: `src/editor/propsForm/PropsForm.tsx` + `describePropsSchema` (`src/catalog/zodIntrospect.ts`) — контролы switch/select/text/number/json/asset, required/optional, дефолты, черновики полей. Живая zod-схема компонента доступна на странице бесплатно: `loadCustomComponents` возвращает `definitions[name].props` (настоящий `z.ZodType` из бандла) — тот же объект, на котором форма уже работает в инспекторе редактора.

Отклонено: новый рендерер форм из `propsJsonSchema` (JSON Schema из манифеста). Это второй параллельный механизм с собственными багами; JSON Schema беднее для контролов (нет живого safeParse), а бандл на странице превью всё равно загружен.

### D3. Docs-таблица props — из `propsJsonSchema`, не из zod-интроспекции

`describePropsSchema` теряет `.describe()` и min/max; в `propsJsonSchema` (манифест, `CatalogComponent.propsJsonSchema`) есть descriptions, enum'ы, дефолты, required. Поэтому: контролы — из zod (D2), props-таблица докстраницы — из `propsJsonSchema`. Источники не конфликтуют: оба производные одной `definition.props`.

### D4. Роут `/library/c/:componentId`, версия — query `?v=<n>`

Отдельный сегмент `c/` исключает коллизии с будущими под-роутами Library. Без `?v` — активная published-версия из манифеста; `?v=<n>` — конкретная (нужно для сравнения версий и ссылок из статусов). Страница — внутри `Layout` (обычная навигация app-шелла). Deep-link становится шарябельным URL компонента.

### D5. Невалидные значения контролов не доходят до рендера

Коммит значения поля проходит `props.safeParse` живой zod-схемой; при ошибке — inline-ошибка у поля, превью продолжает показывать последний валидный набор props. Это же правило защищает от рассинхрона примеров: пресет из `examples` перед применением тоже прогоняется через safeParse (примеры валидировались на publish, но против схемы своей ревизии — при `?v=` другой версии могут не сойтись).

### D6. Тема — латест темы дизайн-системы компонента, как в capture

Идентично `CaptureComponent.loadComponent`: `getDesignSystemById(designSystem)` → `{tokens, fonts, icons}` → `ThemeStyle` + `SurfaceSpacingScope`. Компоненты не пинуются к версии темы (существующая политика). Переключатель light/dark меняет только фон/хром области превью (как `?theme=` в capture меняет только хром) — у custom-тем нет dark-варианта токенов.

## 3. Декомпозиция работ и file ownership

Задачи W1–W2 независимы (разные зоны), W3–W4 зависят от обеих. Каждая задача — отдельный `--fresh` диспатч Codex `--write --effort medium`; коммиты делает оркестратор после независимой проверки done-критериев.

### W1. Вынос ядра PropsForm из редактора

**Владение:** `src/editor/propsForm/**`, новый `src/propsForm/**` (shared), `src/catalog/zodIntrospect.ts` (только чтение, изменения запрещены).

Сейчас `PropsForm` сцеплена с редактором: `validateElementProps` (динамические `$`-значения и state прототипа), `DocEpochContext` (undo/redo), `uploadAsset`/`EditorAsset` (ассеты редактора). Выделить ядро полевых контролов в `src/propsForm/`:

- `FieldControl` (рендер одного поля по `PropField`: switch/select/text/number/json) и общий каркас формы с черновиками/ошибками — без editor-зависимостей;
- инъекция специфики через props: `validate(candidate) => FieldErrors` (редактор передаёт `validateElementProps`-обёртку, витрина — zod `safeParse`-обёртку), опциональный `renderAssetField` (редактор передаёт свой `AssetField`, витрина — нет: asset-props показываются text/json-контролом), опциональный `epoch` (редактор передаёт `DocEpochContext`, витрина — константу);
- `src/editor/propsForm/PropsForm.tsx` становится тонкой обёрткой над ядром с прежним публичным API — **поведение инспектора редактора не меняется**.

Done-критерии:
- `npm run verify` зелёный; существующие тесты propsForm/инспектора проходят без изменения ожиданий (правки тестов — только пути импортов);
- в `src/propsForm/**` нет импортов из `src/editor/**` и `src/api/client` (проверяется grep'ом);
- ручная проверка инспектора в `/p/:id/edit` (typecheck + существующие e2e редактора).

### W2. Docs-модель: рендер `propsJsonSchema`/событий/слотов (чистые компоненты)

**Владение:** новый `src/library/componentDocs/**`.

Чистые presentational-компоненты без загрузки данных (данные придут из W3):

- `PropsTable({ schema })` — таблица из JSON Schema объекта: имя, тип (включая enum-значения и вложенные объекты/массивы — для сложных типов сворачиваемый JSON-вид), default, required, description. Оборонительно к неполной схеме (schema может отсутствовать: extraction best-effort);
- `EventsSection({ events, eventPayloads })` — список событий, payload-схема как компактный JSON-вид;
- `SlotsSection({ slots })`, `MetaSection` (description, atomicLevel, capabilities, designSystem);
- `SourceView({ source })` — readonly `<pre>` c TSX (без подсветки в MVP; подсветка — отдельное улучшение).

Done-критерии:
- юнит-тесты на `PropsTable`: enum с default, optional со `description`, вложенный объект, отсутствующая/пустая схема (не падает);
- `npm run verify` зелёный; нет обращений к API из этих модулей.

### W3. Страница компонента `/library/c/:componentId`

**Владение:** новый `src/library/componentPage/**`, `src/app/routes.tsx` (только добавление роута), `src/api/client.ts` (только при необходимости — новых endpoints не ожидается).

Сборка страницы из W1+W2:

- загрузка: манифест (`getCatalogManifest`) → entry компонента; `getComponentMeta` (версии/статусы/figma); `getComponentVersion(id, v)` (source, examples выбранной версии); `loadCustomComponents([...])` — бандл выбранной версии; тема — как в D6;
- разметка: превью-область (инлайн-рендер по образцу `LoadedComponentCapture`: `SurfaceSpacingScope` → `ThemeStyle` → `CaptureSurface` c single-element `toRuntimeSpec`) + панель контролов (ядро из W1, схема — живой zod из бандла) + чипы пресетов из `examples` (+ `default` = дефолты схемы) + переключатель светлый/тёмный фон превью + селектор версии (`?v=`) + бейдж статуса версии (`statusBadge.ts`);
- вкладки: «Компонент» (превью+контролы), «Документация» (W2-компоненты на данных манифеста/версии), «Код» (`SourceView`);
- состояния ошибок: компонент не найден / версия не найдена / бандл не загрузился (ошибка вместо превью, docs-вкладка при этом работает — ей бандл не нужен) / схема не объект (fallback: JSON-редактор всего объекта props, как `JsonWholeProps` в редакторе);
- слоты: для компонентов со slots превью рендерит плейсхолдер-контент в `default`-слот (простой текстовый блок), чтобы контейнерные компоненты (yp-box) не схлопывались в ноль.

Done-критерии:
- роут работает в dev: страница yp-box (или fixture-компонента) показывает превью, изменение enum-контрола видимо меняет рендер без перезагрузки страницы;
- невалидный ввод (число вне min/max у typed-events-stars) показывает ошибку поля и не ломает превью;
- `?v=` переключает версию, несуществующие id/версии дают человеческие ошибки;
- `npm run verify` зелёный.

### W4. Интеграция в Library

**Владение:** `src/library/LibraryPage.tsx`, `src/library/libraryModel.ts`, `src/app/strings/**` (строки).

- Карточка custom-компонента (`ComponentMetadata`) получает ссылку «Открыть страницу компонента» → `/library/c/:id`; сама карточка остаётся компакт-обзором (превью-iframe в ней не трогаем — быстрый обзор в master-detail);
- выбор компонента в nav по-прежнему открывает карточку; страница — по явному переходу (双-клик/ссылка). Никаких изменений builtin-веток.

Done-критерии: переход из Library на страницу и обратно (браузерная навигация) работает; `npm run verify` зелёный.

### W5. E2E и приёмка

**Владение:** `e2e/**` (новый spec), при необходимости — seed-скрипт e2e-данных.

- e2e-сценарий (vite+API dev, `.e2e-data/dev`): в setup опубликовать fixture-компонент через API (переиспользовать `server/fixtures/typed-events-stars.tsx` — у него типизированные события и числовой prop с min/max), открыть `/library/c/<id>`, дождаться превью, поменять `value` контролом → проверить видимое изменение, ввести невалидное значение → проверить ошибку поля, открыть вкладки «Документация» (props-таблица содержит `value`) и «Код» (содержит `z.strictObject`);
- финальная приёмка оркестратором: `npm run verify` + `npm run e2e` + runtime-прогон по `.claude/skills/verify/SKILL.md` (включая регресс инспектора редактора после W1).

## 4. Риски

| Риск | Митигция |
|---|---|
| Вынос ядра PropsForm регрессит инспектор редактора | W1 сохраняет публичный API и тесты редактора; e2e редактора в приёмке W5 |
| Тема DS конфликтует со стилями app-шелла на новой странице | Прецедент плеера/редактора; `ThemeStyle` restore-on-unmount; проверка в W3 runtime |
| `propsJsonSchema` отсутствует/неполна (extraction best-effort) | W2: PropsTable оборонительна; контролы не зависят от неё (живой zod) |
| Компонент с обязательными props без дефолтов рендерится с `{}` и падает | Начальные props = `example` версии (если есть) → иначе дефолты схемы → иначе `{}`; ошибка рендера ловится и показывается вместо превью (error boundary вокруг превью-области) |
| `examples` версии не проходят схему другой версии при `?v=` | D5: safeParse перед применением пресета, невалидный пресет дизейблится с подсказкой |
| Крупный компонент/бандл тормозит страницу | Бандлы immutable-кешируются; лоадер `loadCustomComponents` кеширует promise по URL |

## 5. Верификация (итог)

1. `npm run verify` — после каждой волны.
2. `npm run e2e` — после W5.
3. Runtime-прогон `/verify`: Library → карточка → страница компонента → контролы/доки/код → регресс редактора и плеера.
