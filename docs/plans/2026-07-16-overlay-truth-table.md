# Overlay StageViewport truth table (W4b entry-gate)

Нормативная модель — план `2026-07-15-spacing-layout.md` §D5: четыре бокса `ClipViewport` / `StageViewport` (native-координаты, внутри transform chain контента) / `ContentScroller` / portal root. Overlay — viewport-sticky: якорится к StageViewport и не двигается относительно него; overlay-portal монтируется в `stageHostRef`, предоставленный поверхностью. На canvas-экранах Overlay — третий упорядоченный слой `CanvasLayers` (content < hotspots < overlay). Desktop-экран без canvas — Overlay запрещён (validate error + runtime-правило device-switcher).

## Инварианты (проверяются тестами W4b/W4c)

1. **stageHostRef** каждой поверхности указывает на узел StageViewport из таблицы (или на его прямой relative-контейнер той же геометрии); overlay-portal монтируется в него; НИКАКИХ обёрток вокруг legacy flow — containing block и margin collapsing существующих элементов не меняются (DOM-тест: absolute-ребёнок контента находит тот же offsetParent до и после подключения Overlay-инфраструктуры).
2. Overlay в native-координатах: `inset` из `var(--eui-space-*)` применяется до transform, т.е. масштабируется вместе с контентом (тесты editor scale≠1, gallery двойной scale, player zoom).
3. `top`/`bottom` placement — stretch на ширину StageViewport минус горизонтальные insets; углы/center — shrink-to-fit.
4. Порядок canvas-сплитов: `splitHostPrimitives` → `splitCanvas`; action runtime получает полный `tree.spec`.
5. Device-switcher runtime-правило (§3): disabled + сброс override; тесты mobile/tablet→desktop, canvas→non-canvas, смена экранов и версий.
6. `SurfaceSpacingScope` подключается на каждый StageViewport-узел (W4b: player/present/capture уже есть на capture; W4c: editor/CJM/gallery/stories) с resolved-шкалой pinned-темы поверхности.
7. Storybook-story Overlay не меняет поведение голого Renderer для остальных историй.
