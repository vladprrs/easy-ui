# CJM-вид прототипа (второй режим просмотра рядом с плеером)

## Context

У прототипа сейчас единственный режим просмотра — интерактивный плеер (один экран, навигация actions). Нужен второй режим — **CJM-вид**: все экраны прототипа выложены горизонтальной лентой слева направо со стрелками-коннекторами, каждый экран живо отрендерен **в своём состоянии**, с заголовком и авторским комментарием под экраном. Переключение между плеером и CJM в обе стороны.

Решения, согласованные с пользователем:
- Подписи — **новые поля в JSON прототипа** (per-screen `note`).
- Состояние экрана в CJM — `doc.state` + **per-screen `stateOverrides`** (deep merge), чтобы один экран можно было показать пустым/заполненным/с ошибкой.
- Экраны в CJM **статичны** (живой рендер, но клики перехвачены); клик по карточке открывает этот экран в плеере.
- Раскладка — **горизонтальная лента** с коннекторами.

Ключевые проверенные факты:
- `screenSchema` в `src/prototype/schema.ts` — `strictObject { id, name, canvas?, spec }`; сервер (`server/routes/prototypes.ts`, `server/seed.ts`), `scripts/validate-prototypes.ts` импортируют эту же схему — правка одна, дублей нет.
- `PlayerNavigationProvider` (`src/player/navigation.tsx:44-51`) на bootstrap принудительно редиректит на `/s/:screenId` → CJM **нельзя** вкладывать под `PlayerShell`, нужен сиблинг-роут `CjmShell`.
- Обычный `<Link to="/p/x/s/y">` без location state бутстрапит плеер ровно на этом экране (`target = screenId ?? startScreen`) — «клик по тайлу → плеер на этом экране» получается бесплатно.
- Паттерн изолированного рендера есть: `SmokeRenderer` (`src/smoke/SmokeSpec.tsx`) — свой `JSONUIProvider` + `Renderer` с собственным `initialState`; runtime из `createPlayerRuntime` шарится между N провайдерами.
- `splitCanvasSpec` сейчас приватен в `src/player/ScreenView.tsx`; `routeBase` вычисляется только в `PlayerShell` и не пробрасывается через `PlayerOutletContext`.

## Шаги

### 1. Схема: новые поля экрана
`src/prototype/schema.ts` — в `screenSchema` добавить:
```ts
note: z.string().min(1).optional(),                           // подпись под тайлом в CJM
stateOverrides: z.record(z.string(), z.unknown()).optional(), // deep merge поверх doc.state
```
Плоские поля (не sub-object `cjm:{}`): `note` пригодится и вне CJM. Оба optional — все существующие `prototypes/*.json` остаются валидны.

**Новый `src/prototype/stateOverrides.ts`**: `mergeScreenState(base, override?)` — deep merge: рекурсия когда обе стороны plain-объекты; скаляры/массивы/несовпадение типов — override заменяет целиком; `undefined` в override игнорируется. Deep merge, потому что state вложенный (`form.name`, `cart.count`) — shallow заставил бы переписывать целые поддеревья.

**`src/prototype/validate.ts`**: для `stateOverrides` — error на зарезервированных top-level ключах (`currentScreen`, `navStack`, `_viewer`), warning на ключах, которых нет в `doc.state` (в духе существующего warning про `$state`).

**`docs/prototype-format.md`**: описать `note`, `stateOverrides`, семантику deep merge (массивы заменяются), порядок экранов в CJM = порядок массива `screens`.

### 2. Извлечь общую загрузку из PlayerShell (рефакторинг без изменения поведения)
`src/player/PlayerShell.tsx` (+ новый `src/player/usePrototype.ts`):
- Экспортировать `MissingPrototype`, `LoadError`.
- Хук `usePrototypeWithComponents(protoId?, version?)` — оборачивает оба `useApi`-вызова (draft/version + `loadCustomComponents`), возвращает discriminated status и `runtimeKey` (`${doc.id}:${revision}:${componentManifestHash}:${doc.designSystem}`). Deps-массивы `[protoId, version]` и `[loaded.componentManifestHash]` скопировать дословно.
- `splitCanvasSpec` вынести из `ScreenView.tsx` в новый `src/player/canvasSpec.ts`.
- В `PlayerOutletContext` добавить `routeBase: string` (нужно тумблеру в сайдбаре).

### 3. Роуты
`src/app/routes.tsx` — сиблинги, не вложенные:
```tsx
<Route path="p/:protoId/cjm" element={<CjmShell />} />
<Route path="p/:protoId/v/:version/cjm" element={<CjmShell />} />
```

### 4. CJM-компоненты — новый каталог `src/cjm/`
- **`CjmShell.tsx`** — роут-элемент: `usePrototypeWithComponents` + `MissingPrototype`/`LoadError`; при ready → `<CjmView doc custom runtimeKey routeBase />` (routeBase считается как в `PlayerShell.tsx:75`).
- **`CjmView.tsx`** — один runtime с no-op nav deps (`createPlayerRuntime({navigate(){}, back(){}, openUrl(){}, restart(){}}, custom, doc.designSystem)`); шапка: `doc.name`, `doc.description`, ссылка «Открыть плеер» → `routeBase`; лента `<ol className="flex items-start gap-0 overflow-x-auto p-6">` по `doc.screens` **в порядке массива**; между тайлами коннектор-стрелка (`<li aria-hidden>`, простой flexbox/SVG).
- **`CjmScreenTile.tsx`** — карточка экрана:
  - заголовок `screen.name`, подпись `screen.note` под фреймом (опциональна);
  - живой рендер в изолированном состоянии:
    ```tsx
    <JSONUIProvider key={`${runtimeKey}:${screen.id}`} registry={runtime.registry}
        handlers={runtime.handlers}
        initialState={mergeScreenState(doc.state, screen.stateOverrides)}>
      <Renderer registry={runtime.registry} spec={contentSpec} />
    </JSONUIProvider>
    ```
  - canvas-экраны: `splitCanvasSpec(toRuntimeSpec(spec)).content`, хотспоты не рендерить (гард на `content === null` — плейсхолдер); обычные экраны: `toRuntimeSpec(spec)`;
  - масштабирование — **не** переиспользовать `DeviceFrame` (в нём тумблер устройств и ResizeObserver): мини-`CjmFrame` — `TILE_WIDTH = 280`, nativeWidth = `canvas?.width ?? {mobile:390, tablet:834, desktop:1280}`, `transform: scale(TILE_WIDTH/nativeWidth)` + `overflow:hidden`, maxHeight ≈ 420px для экранов без известной высоты;
  - статичность: `inert` на обёртке контента + полнотайловый `<Link to={buildPlayerPath(routeBase, screen.id)} className="absolute inset-0" aria-label=... />` (`buildPlayerPath` из `src/player/navigation.tsx`);
  - per-tile error boundary (аналог `ScreenErrorBoundary` без Restart) — сломанный экран не валит всю ленту.

### 5. Тумблер в плеере

### 6. Галерея
`src/gallery/GalleryPage.tsx` — рядом с pill «Published vN» добавить pill-ссылку `/p/${id}/cjm`.

### 7. Демо-фикстура

После — `npm run validate:prototypes`. Схема (шаг 1) должна лечь раньше: `server/seed.ts` перепарсит файлы.

## Порядок

1 (схема+merge+validate+docs) → 2 (извлечение из PlayerShell, тесты зелёные) → 3–4 (роуты+CJM) → 5–6 (тумблеры/ссылки) → 7 (фикстура) → 8 (тесты, verify).

## Риски

- **Порталы**: Dialog/Drawer/Popover рендерятся вне inert-обёртки — spec с открытым по умолчанию диалогом «сбежит» из тайла. Приемлемое ограничение v1 (диалоги по умолчанию закрыты); при необходимости — CJM-scoped CSS.
- **N провайдеров**: полный `JSONUIProvider`+`Renderer` на тайл; при лимите 500 элементов/экран лента из 5–10 экранов ок; ленивый маунт — вне скоупа.
- **Desktop-экраны без canvas** не имеют внутренней высоты — виртуальный вьюпорт 1280×800 + maxHeight, высокие экраны обрезаются (тайл ведёт в настоящий плеер).
