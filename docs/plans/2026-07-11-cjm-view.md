# CJM-вид прототипа (второй режим просмотра рядом с плеером)

Ревизия 3 — после двух раундов адверсариального ревью Codex (тред 019f50ea-97a0-75b2-8778-68a6c1b00ddf); триаж находок в конце файла.

## Context

У прототипа сейчас единственный режим просмотра — интерактивный плеер (один экран, навигация actions). Нужен второй режим — **CJM-вид**: все экраны прототипа выложены горизонтальной лентой слева направо со стрелками-коннекторами, каждый экран живо отрендерен **в своём состоянии**, с заголовком и авторским комментарием под экраном. Переключение между плеером и CJM в обе стороны.

Решения, согласованные с пользователем:
- Подписи — **новые поля в JSON прототипа** (per-screen `note`).
- Состояние экрана в CJM — `doc.state` + **per-screen `stateOverrides`** (deep merge), чтобы один экран можно было показать пустым/заполненным/с ошибкой.
- Экраны в CJM **статичны** (живой рендер, но клики перехвачены); клик по карточке открывает этот экран в плеере.
- Раскладка — **горизонтальная лента** с коннекторами.

Ключевые проверенные факты:
- `screenSchema` в `src/prototype/schema.ts` — `strictObject { id, name, canvas?, spec }`; сервер (`server/routes/prototypes.ts`, `server/seed.ts`), `scripts/validate-prototypes.ts` импортируют эту же схему — правка одна, дублей нет (но нужны schema-тесты и round-trip тесты сервера).
- `PlayerNavigationProvider` (`src/player/navigation.tsx:44-51`) на bootstrap редиректит на `/s/:screenId`; вложение CJM под текущий `PlayerShell` неудобно, потому что тот целиком оборачивает `Outlet` навигационным и JSONUI-провайдерами. Сиблинг-роут `CjmShell` — **сознательно простой выбор**, а не техническая невозможность.
- Обычный `<Link to="/p/x/s/y">` без location state бутстрапит плеер на этом экране (`target = screenId ?? startScreen`); редирект идёт через `replace`, поэтому browser Back возвращает в CJM (проверить e2e). Открывается **новая** player-сессия (`flowDepth: 0`, Back в плеере неактивен) — это ожидаемое поведение.
- Изоляция состояния: каждый `JSONUIProvider` создаёт собственный store (`StateProvider` в `@json-render/react@0.19.0`) — json-render-state тайлов изолирован; registry/handlers шарятся безопасно. Гарантия честная: изолирован именно store, произвольные side effects кастомных компонентов — нет.
- `splitCanvasSpec` сейчас приватен в `src/player/ScreenView.tsx`.

## Шаги

### 1. Схема: новые поля экрана
`src/prototype/schema.ts` — в `screenSchema` добавить:
```ts
note: z.string().trim().min(1).max(500).optional(),      // подпись под тайлом в CJM
stateOverrides: z.record(z.string(), jsonValueSchema).optional(), // deep merge поверх doc.state
```
Ввести рекурсивную `jsonValueSchema` (string | number | boolean | null | массив | record) и применить её **и к `doc.state`, и к `stateOverrides`** — контракт «JSON-only» обеспечивается схемой, а не только транспортом (существующие JSON-файлы валидны по построению).
Плоские поля (не sub-object `cjm:{}`). Оба optional — существующие `prototypes/*.json` валидны. В `docs/prototype-format.md` зафиксировать: v1 эволюционирует аддитивными optional-полями.

**Новый `src/prototype/stateOverrides.ts`**: `mergeScreenState(base, override?)` — **безопасный** deep merge со строгим контрактом:
- только JSON-значения; рекурсия когда обе стороны plain-объекты; скаляры/массивы/несовпадение типов — override заменяет целиком; `null` — обычная замена; пустой объект `{}` не удаляет базовые поля; удаления полей в v1 нет;
- обходить только own enumerable string keys; ключи `__proto__`, `prototype`, `constructor` **запрещены на любой глубине** (пропускать при merge + error в validate);
- результат строить через spread/`Object.fromEntries`, не мутировать ни `doc.state`, ни override; результат не должен разделять object-ссылки с override;
- depth limit (например 32): объект глубже лимита **не вставляется целиком** (иначе внутрь пролезут запрещённые ключи глубже лимита) — merge пропускает слишком глубокое поддерево (остаётся base) и это же ловится ошибкой в validate; скан запрещённых ключей обходит **весь** override независимо от depth limit;
- тесты: pollution (`__proto__` и др., в т.ч. глубже depth limit), inherited properties, `null`, массивы, type replacement, немутация входов, отсутствие разделяемых ссылок.

**`src/prototype/validate.ts`**:
- error на зарезервированных top-level ключах override (`currentScreen`, `navStack`, `_viewer`), на запрещённых ключах любой глубины и на превышении depth limit;
- **effective-state валидация**: для каждого экрана считать `effectiveState = mergeScreenState(doc.state, screen.stateOverrides)` и проверять `$state`/`$bindState`/`visible`/state-пути этого экрана против него, а не против `doc.state` (иначе ложные warnings для путей, существующих только в override). Отдельный warning «top-level ключ отсутствует в doc.state» не делать.

**`docs/prototype-format.md`**: `note`, `stateOverrides`, контракт merge (JSON-only, массивы заменяются, `{}` не удаляет, запрещённые ключи), порядок экранов в CJM = порядок массива `screens`, аддитивная эволюция v1.

### 2. Извлечь общую загрузку из PlayerShell (рефакторинг без изменения поведения)
Вместо единого хука (нельзя условно вызвать второй `useApi`, и один discriminated status теряет раздельные retry) — **staged render-prop компонент** `src/player/PrototypeLoader.tsx`:
```tsx
<PrototypeLoader protoId={protoId} version={version}>
  {({ loaded, custom, runtimeKey, routeBase }) => ...}
</PrototypeLoader>
```
- Внутри — та же двухступенчатая структура, что сейчас в `PlayerShell`/`ReadyPlayer`: первый `useApi` (draft/version, deps `[protoId, version]`), при ready — дочерний компонент со вторым `useApi` (`loadCustomComponents`, deps `[loaded.componentManifestHash]`); раздельные retry сохранены; `MissingPrototype`/`LoadError` рендерятся самим лоадером; `key` дочерней стадии — по revision (как сейчас `key={runtimeKey}` в `ReadyPlayer`), чтобы component-load-состояние не мигрировало между ревизиями.
- `runtimeKey` = `${doc.id}:${revision}:${componentManifestHash}:${doc.designSystem}`, `routeBase` — как в `PlayerShell.tsx:75`.
- `PlayerShell` переписать поверх `PrototypeLoader` (поведение и тесты не меняются).
- `splitCanvasSpec` вынести из `ScreenView.tsx` в `src/player/canvasSpec.ts` (+ тест на hotspot-root → `content === null`).
- `PlayerOutletContext` **не менять** (упрощение по ревью).

### 4. CJM-компоненты — новый каталог `src/cjm/`
- **`CjmShell.tsx`** — роут-элемент поверх `PrototypeLoader`; при ready → `<CjmView doc custom runtimeKey routeBase />`.
- **`CjmView.tsx`**:
  - один runtime с no-op nav deps: `createPlayerRuntime({navigate(){}, back(){}, openUrl(){}, restart(){}}, custom, doc.designSystem)`;
  - шапка: `doc.name`, `doc.description` (условный рендер — блока нет, если описания нет), ссылка «Открыть плеер» → `routeBase` (откроет `startScreen` — задокументировать);
  - лента `<ol>` по `doc.screens` в порядке массива; **один `<li>` на экран**, коннектор-стрелка — вложенный `aria-hidden` SVG/pseudo-element внутри `li`, не отдельный list item;
  - **перехват порталов (blocker №1)** — CSS-гашение НЕ решает проблему: открытый Radix `Dialog` даже с `display:none` на контенте оставляет modal side effects (`hideOthers` → `aria-hidden` на остальной странице, `FocusScope trapped`, глобальные focus guards, `document.body.style.pointerEvents="none"`, `RemoveScroll`); у Vaul `Drawer` аналогично. Решение — **CJM-политика реестра**: в `CjmView` после `createPlayerRuntime` оборачивать записи реестра portal-типов так, чтобы модальные примитивы не монтировались открытыми: `Dialog`/`Drawer` получают принудительно `open=false` (контент не монтируется); `Select`/`Popover`/`DropdownMenu`/`Tooltip` открываются только от interaction, который блокирует `inert`, — их достаточно тоже принудительно держать закрытыми для симметрии. **Исполнитель обязан изучить фактическое API registry `@json-render/react` и компоненты `@json-render/shadcn` в node_modules** (как устроены записи реестра и props Dialog/Drawer), не угадывать. `ConfirmationDialogManager` — НЕ портал (обычный fixed div внутри `JSONUIProvider`), в CJM он появляется только от action, которые блокирует inert, — в политику не входит, но покрывается тестом «inert-тайл не может инициировать confirm action». Регрессионный тест (browser): экран со `stateOverrides`, «открывающим» Dialog, — `document.body.style.pointerEvents !== "none"`, overlay-ссылки тайлов реально кликаются, лента не получает `aria-hidden`, `Tab` не попадает в focus guards, body прокручивается.
- **`CjmScreenTile.tsx`**:
  - заголовок `screen.name`, подпись `screen.note` (опциональна);
  - изолированный рендер: свой `JSONUIProvider` с `initialState={mergeScreenState(doc.state, screen.stateOverrides)}`; canvas-экраны — `splitCanvasSpec(toRuntimeSpec(spec)).content` без хотспотов (гард `content === null` → плейсхолдер); обычные — `toRuntimeSpec(spec)`;
  - **геометрия (blocker №3)** — двухслойный `CjmFrame`: `transform: scale()` не меняет layout-размеры, поэтому внешний viewport задаёт реальные размеры в потоке:
    - внутренний native viewport: `width: nativeWidth` (`canvas?.width ?? {mobile:390, tablet:834, desktop:1280}`), для canvas — `height: canvas.height`; `transform: scale(TILE_WIDTH/nativeWidth)`, `transform-origin: top left`;
    - внешний viewport: `width: TILE_WIDTH (280)`, `height`: для canvas/известной высоты — `nativeHeight * scale`; для non-canvas — измерять native `scrollHeight` внутреннего слоя через `ResizeObserver` (observer видит немасштабированный размер — умножать на scale) и ставить `min(measuredHeight * scale, CAP≈420)`, `overflow: hidden`; задать fallback-высоту до первого callback и пересчитывать при смене spec/runtimeKey;
  - статичность: `inert` на content-обёртке + сиблинг-`<Link to={buildPlayerPath(routeBase, screen.id)} className="absolute inset-0">` с видимым focus-стилем и `aria-label` вида «Открыть экран “{name}” прототипа “{doc.name}” в плеере»; Link не вкладывать в другой интерактивный элемент; на published-роуте ссылка сохраняет `/v/N`;
  - per-tile error boundary; **key = `${runtimeKey}:${screen.id}`** (сбрасывается при новой ревизии, не залипает в error).
- Storybook: лёгкая стори для `CjmFrame` (mobile/tablet/desktop, canvas, длинный контент) вне catalog-drift.

## Риски

- **Порталы** — закрываются CJM-политикой реестра (см. §4); остаточный риск: кастомный компонент с собственным порталом/модалкой — не изолируется, задокументировать как ограничение.
- **Кастомные компоненты** с собственным локальным состоянием/браузерными side effects не изолируются store-изоляцией — честно задокументировать в format-доке.
- **N провайдеров**: полный `JSONUIProvider`+`Renderer` на тайл; лента 5–10 экранов ок; ленивый маунт — вне скоупа.
- **Non-canvas desktop-экраны**: высота через измерение + CAP, длинные экраны обрезаются (тайл ведёт в настоящий плеер).

## Процесс (workflow из CLAUDE.md)

Ревизия 2 отправляется на повторный раунд ревью (`--resume` того же треда). После снятия блокирующих возражений — декомпозиция на `--fresh --write --effort medium` задачи с file ownership (schema/prototype ⟂ PrototypeLoader-рефакторинг ⟂ cjm ⟂ gallery/e2e), независимая верификация done-критериев, поэтапные коммиты.
