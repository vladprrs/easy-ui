# План v4: UX-улучшения пользовательского UI — плеер и редактор в приоритете (2026-07-13)

Источник: multi-agent UX-ревью от 2026-07-13 — шесть зонных ревьюеров (плеер, редактор, галерея/библиотека, CJM/visual, app-шелл/a11y, продуктовая стратегия), каждый прочитал код своей зоны и прогнал живое приложение через Playwright (137 скриншотов, эмуляция ошибок API и узких viewport), плюс критик полноты с живой перепроверкой спорных находок. Итог: 101 находка (2 подтверждённых P0, ~30 P1), 15 пропущенных областей, 11 спорных пунктов. Ревью плана (Codex gpt-5.6-sol, max): раунд 1 — 10 blocker / 20 major / 4 minor, все приняты (2 с модификацией); раунд 2 — 10 blocker / 12 major / 2 minor, все приняты; раунд 3 (контрольный) — 5 blocker / 4 major, все приняты (точечные DAG/ownership-правки, архитектура подтверждена). Триаж всех раундов в конце. Этот план **замещает** `docs/plans/2026-07-10-flow-state-persist-share.md` (файл помечен superseded).

## Ключевые факты ревью

- **Главный сценарий продукта — «показать прототип заказчику» — сейчас проваливается на первом же показе.** На /p/* друг под другом ~200px хрома (белый app-header + тёмный header плеера + строка device-чипов); `DeviceFrame` масштабирует только по ширине (`src/player/DeviceFrame.tsx:29`: `scale = min(1, availW/w)`), фрейм 390×844 не влезает по высоте даже в 1440×900 — CTA прототипа всегда за фолдом. Заказчик видит всю оснастку: навигацию Gallery/Library/Visual/**Debug**, сайдбар со списком всех экранов (спойлер финала флоу), переключатель устройств.
- **Флагманское демо встречает битым визуалом и противоречивым стейтом.** `prototypes/checkout.json` ссылается на несуществующий `/images/sneakers.jpg` (SPA-fallback отдаёт 200 text/html → сломанный `<img>` во всех вью). Критик подтвердил: `cart-total` — захардкоженный текст «Итого: 7 990 ₽ · товаров: ${/cart/count}» → при deep-link/reload экран показывает «Корзина пуста» и «Итого: 7 990 ₽ · товаров: 0» одновременно, «Оформить» активна при пустой корзине.
- **Сервер сильнее UI ровно в точке ценности.** `publishPrototype`/`listPrototypeVersions`/`listPrototypeRevisions`/`restorePrototype`/`createPrototype` уже есть в `src/api/client.ts:137-157`, но ни одна страница их не вызывает: на живой базе у всех прототипов `latestVersion=null`. Публикация, версии, история, откат, создание — curl-only. Шаринг наружу требует выдать BasicAuth от всего workspace (`server/main.ts`).
- **Редактор (шаг 1) архитектурно пригоден для конструктора, но UX-дыры делают его опасным.** Лента экранов без капа высоты (`src/editor/EditorScreenStrip.tsx:26-36`) → на 1440×900 канвас+инспектор видны полоской ~80px (критик: на 1024×700 скроллится, severity P1). Тихая потеря правок при SPA-навигации (beforeunload не ловит react-router), нет undo/redo, `Number('') === 0` молча коммитит 0 (`src/editor/propsForm/PropsForm.tsx:80`), события элементов в инспекторе невидимы, восстановление после 409 — тупик «скопируй JSON вручную». Ревью нашло латентную потерю данных: клиентский `PrototypeDraft` отбрасывает `figma`/assets, каждое editor-save создаёт ревизию без provenance (`src/api/client.ts:59`).
- **Reload/deep-link молча сбрасывает стейт флоу** (`src/player/navigation.tsx:48-55`) — получатель ссылки на середину флоу видит противоречивый экран без объяснения. Станет массовым со share-ссылками.
- **404 версии выдаёт «Prototype not found»** — сервер сам не различает случаи: `server/repos/prototypes.ts:210` возвращает общий `not_found`.
- **Смесь RU/EN на каждой поверхности** (все 6 зон); `document.title` всегда «easy-ui», favicon 404.
- **Что уже хорошо (не сломать):** deep-links экранов + браузерная история + sessionNonce/stale-гейт; per-screen ErrorBoundary; Hotspot — настоящая `<button>` с aria; PropsForm из zod-схем; editorReducer/docMutations — чистые и иммутабельные; CJM-тайлы с живыми компонентами; graceful degradation Library; образцовая API-поверхность.
- **Пропуски ревью, влияющие на план (критик):** мобильный браузер заказчика (≤430px) — плеер непригоден; desktop-прототип в плеере — белый лист без min-height; custom DS в редакторе/плеере не проверялись; редактор не открывался на composition-фичах; длинные названия и 20+ экранов не проверены; UI управления ассетами не существует.
- **Факты из ревью плана:** `useBlocker` требует data-router (`BrowserRouter` в `src/app/main.tsx:9` не подходит — хук падает в рендере); `location.state` живёт в `window.history.state.usr` и **переживает reload** — один `entryReason` не отличит reload от flow-навигации; `server/seed.ts` применяет фикстуру один раз по `seed_log` и запрещает custom components; повторный PUT прототипа — осознанный checkpoint (репин бандлов/темы, CAS, message), не no-op; share-страница тянет субресурсы, которым нельзя передать токен-заголовок; Vite-хэши статики меняются при redeploy — allowlist статики нельзя замораживать при создании grant; `upsertReference` заменяет asset_id, `visual_runs` не хранит baseline-asset, а FK каскадно удаляет runs (`server/migrations.ts:107`); visual поддерживает и prototype-screen, и component references; visual-fingerprint содержит явные width/height и «не изменился» не доказывает неизменность пикселей; `/api/assets` — только POST; Image приходит из `@json-render/shadcn`; `__euiKey` инжектится только custom-элементам (`src/prototype/runtimeSpec.ts:105`) — для builtin production-маркера нет; canvas-превью редактора/CJM зовут legacy `stripSpecEvents`/`splitCanvasSpec` вместо RuntimeTree-операций; сервер принимает `figma:null` как очистку, но контракт разрешает только optional object.

## Сквозные решения (фиксируются до декомпозиции)

1. **Язык продукта — русский, с явной границей.** Локализуется product chrome; authored-контент прототипов и custom-бандлы не трогаются. Словарь — каталог feature-модулей `src/app/strings/` (`common.ts`, `gallery.ts`, `player.ts`, `editor.ts`, `cjm.ts`, `visual.ts`, `library.ts`): константы **и функции** (интерполяция, склонения). Серверные ошибки — через `formatApiError(code, details)` (`common.ts`); это включает и логические правки клиентов API, теряющих `code` (например, `src/visual/api.ts:39`). Задачи последующих волн владеют строками **своих** feature-модулей; строковый feature-модуль — общий файл: задачи, добавляющие строки одного feature, **сериализуются** в DAG волны (правило учтено в DAG W1/W2/W5). Allowlist доменных терминов: CJM, Storybook, API, Basic Auth. EN-рынок вне скоупа.
2. **Единая модель навигации по прототипу — отдельный компонент со slot-контрактом.** `PrototypeChrome` (WF-4): крошка «Галерея / {Имя}» + сегмент «Плеер · CJM · Редактор» + **стабильные слоты** `status` (версия/dirty) и `actions` (Back/Restart/Презентация/Inspector/Заметка/Publish поставляются вью через outlet-context/props). WF-4 фиксирует границу chrome/stage; последующие задачи меняют только тело stage и поставляют экшены в слоты, не трогая сам PrototypeChrome. На version-route сегменты Плеер/CJM сохраняют версию, «Редактор» ведёт в draft-редактор с явным бейджем «черновик» (тихая потеря version-контекста запрещена). Презентация и share рендерятся вне PrototypeChrome.
3. **Единая политика состояния флоу.** Стейт живёт от startScreen. Вход различается `entryReason` (`bootstrap`/`browse`/`flow`) в location state **плюс document-lifetime nonce**: nonce генерируется при каждой загрузке документа, несовпадение nonce в history-state ⇒ это reload ⇒ трактуем как `bootstrap` (location.state переживает reload через `history.state.usr` — одного entryReason недостаточно). `bootstrap` не на startScreen → баннер «Состояние флоу сброшено — Начать сначала?»; `browse` (сайдбар/стрелки) → replace-переход `browseToScreen` вне flowDepth; `flow` — действия прототипа. stateOverrides из CJM в плеер не сеются — тайл получает бейдж «демо-состояние». Query string (в т.ч. ?debug=1) сохраняется всеми переходами.
4. **Один лэйаут-редизайн канваса плеера, а не три патча** (W1-1 поверх WF-3 и WF-4).
5. **Канонический viewport прототипа неприкосновенен.** Модуль `src/designSystems/deviceMetrics.ts` (WF-3) разделяет три понятия: canonical viewport (капчер/валидация; desktop остаётся auto-height — семантика capture не меняется), preview tile sizes (CJM/лента/галерея) и player-only min-height. Декоративный bezel/статусбар — вне канонического viewport. Проверка WF-3 — по точным pixel-хэшам контрольных капчеров, «осознанное обновление baseline» в механической задаче запрещено.
6. **UI закрывает разрыв с API; серверные изменения — только необходимые** (share-сессии W3-3; миграция visual W5-4; коды 404 W0-4). Каждый endpoint: `server/contracts.ts` + перегенерация `server/openapi.json` + happy/error contract-тесты. Миграции строго сериализуются: W0-4 → W3-3 → W5-4.
7. **Превью в галерее — клиентские inert-тайлы с измеримым перф-гейтом.** Фиксированная фикстура 30 карточек (в т.ч. custom DS), cold/warm прогоны, медиана ≥5 запусков, зафиксированный viewport и network-бюджет; гейт: TTI галереи деградирует <20% от baseline без превью. При провале гейта результат W5-3 — карточки без превью (метаданные), а серверные thumbnail-ы оформляются **отдельным пакетом** вне этого плана (не импровизируются внутри задачи).
8. **Save = осознанный checkpoint, без серверной дедупликации** (репин бандлов/темы/ассетов, message). W2-5 сужен до Ctrl+S + документация semantics.
9. **Спорные рекомендации, которые решено НЕ делать:** force-save по умолчанию при 409 (только с трёхсторонним diff, W2-4); disable «Сохранить» при !dirty; авто-граф рёбер CJM (только фикс клиппинга + бейджи статически доказуемых navigate-переходов, динамические — «динамический переход»); скрытие маршрута /debug из прод-сборки (убираем только пункт навигации); очистка figma-provenance из UI (отложена — WF-5 делает только pass-through, nullable-контракт не трогаем).

## Волны исполнения

Каждая задача — свежий Codex-диспатч `--write --effort medium`; владение файлами эксклюзивно в пределах одновременно идущих задач; задачи, делящие файл, соединены в DAG последовательно. Каждая задача владеет **и тест-файлами, которые заведомо ломает** (юниты/e2e), и прогоняет целевые сьюты сама.
