## H1. Токенизация палитры: theme v5 + ABI v4 `color()`

**Проблема.** Палитра не токенизирована. В каталоге **181 уникальная строка цвета**, **32 варианта теней**, ~78 записей радиусов; одинаковые семантические значения записаны по-разному, вплоть до расходящихся fallback одного CSS-токена. Тема v4 содержит **только spacing-токены** — цвета выносить некуда, поэтому фикс-волна согласует литералы лишь точечно и внутри семейств. Конкретные несогласованности из аудита:
- `--shadow-medium`: `yp-screen` использует fallback `0 -8px 24px rgba(0,0,0,.10)`, `yp-block` — `0 8px 24px rgba(0,0,0,.12)` (тот же токен, разные смещение и alpha).
- Градиент Плюса: `yp-plus-progress` даёт `90deg`, остальной Plus-батч — `135deg`; плюс `#edeff2` vs `#f2f3f5` в фоне.
- Топ-цвета по частоте: `#fff` (12), `rgba(255,255,255,.98)` (8), `rgba(0,0,0,.86)`/`rgba(0,0,0,.5)`/`#edeff2`/`#2e2f33` (6), `#1f2023`/`#f3f5f7` (5) — очевидные кандидаты в токены surface/text/fill.
- 72/100 компонентов не импортируют runtime вообще — вся палитра хардкодом.

**Гипотеза.** Ввести **theme v5** с цветовыми токенами (surface/fill/text/separator/shadow/gradient-семейства) поверх существующих spacing-токенов, и **ABI v4** с рантайм-функцией `color(token, fallback?)` (по образцу уже существующей `space()` из ABI v3). Токены — единый источ; компоненты читают `color("text.primary")` вместо литерала. Тени и фирменные градиенты (Плюс, промо) — тоже токены (`shadow.medium`, `gradient.plus`).

**Ожидаемый эффект.** Один канон вместо 181 строки; фикс цвета/тени/градиента — в одном месте, без рассинхрона соседних примитивов; редактор может дать color-picker из токенов; визуальный паритет с реальным Yandex Pay управляется темой, а не правкой 45 файлов. Снимает корневую причину классов C и части D аудита.

**Миграционный путь (breaking-стратегия).**
1. **Сосуществование.** Theme v5 добавляет цветовые токены аддитивно — v4-темы продолжают работать (spacing-токены не трогаются). ABI v4 добавляет `color()` рядом с `space()`/`token()`; старые импорты валидны. Бампается версия темы и ABI, оба поддерживают предыдущие ревизии.
2. **Каталог литералов → токенов.** Из `design-facts-agg.json` собрать реестр «литерал → токен» (canonical-таблица уже в заготовке `design.md`). Мигрировать компоненты семействами: литерал заменяется на `color("…", "<старый-литерал>")` — fallback сохраняет ровно текущий пиксель, поэтому шаг **визуально no-op** и проверяется pixelmatch-диффом = 0.
3. **Миграция доков.** Доков это не касается (цвета внутри компонентов, не в props) — re-pin по регламенту фикс-волны.
4. **design.md** фиксирует токен-канон как контракт для новых компонентов и скилла `yandex-pay`.

**Риски.** Объём (потенциально все 100 компонентов, но безопасно порционно); риск «тихого» изменения оттенка при неверном mapping — снимается тем, что fallback = исходный литерал и дифф обязан быть 0 до момента, когда мы сознательно правим значение токена; тени/градиенты сложнее цветов (multi-stop) — вводить отдельным подэтапом. Реальная экспозиция низкая (аудит: только 2 живых дока опускают опасный проп) — можно катить без спешки.

**Объём: L.** Тема + ABI-рантайм + реестр mapping + помиграционная волна. Реализуемо инкрементально: сначала тема/ABI и топ-20 токенов, дальше по семействам.

## H2. Кросс-семейный канон семантических токенов

**Проблема.** Один семантический цвет записан по-разному в разных семействах — **осознанно отложено из фикс-волны** (конвенция F5.4: выравниваем только внутри семейств). Пример `--text-color-primary`: fallback `#1f2023` (typography), `#000000d8` (form-controls, checkbox/switch), `rgba(0,0,0,.86)` (badges), `#1f1f1f` (feedback-motion), `#111` (часть batch). Аудит зафиксировал **≥7 находок** расхождения только по primary-тексту, плюс аналогичное по «инвертированному белому», secondary, separator.

**Гипотеза.** После введения theme v5 (H1) назначить **канонический fallback** каждому семантическому токену (`text.primary`, `text.secondary`, `text.inverted`, `separator`, …) и свести все семейства к нему. Это отдельная гипотеза, а не часть H1, потому что требует **кросс-семейного решения о «правильном» значении** (какой из `#1f2023`/`#000000d8`/`rgba(0,0,0,.86)` канон — вопрос дизайна, не механики).

**Ожидаемый эффект.** Визуальная консистентность текста/границ по всему каталогу; исчезает целый подкласс minor-находок «расхождение семантического цвета с батчем» (десятки записей). Единый контракт для авторов.

**Миграционный путь.** Зависит от H1 (нужны токены). Решение о каноне — дизайн-ревью (свести 4–5 вариантов к одному эталону Yandex Pay), зафиксировать в design.md. Миграция: заменить расходящиеся fallback на канон **сознательно** (в отличие от H1, здесь пиксель меняется — это ожидаемое улучшение, дифф триажится вручную). Порядок: сначала самые заметные (primary-текст на светлом фоне), затем secondary/tertiary/inverted.

**Риски.** Изменение пикселей на многих экранах разом — нужен прод-baseline и явный триаж диффов (как в W2/W3 фикс-волны). Возможны споры о каноне — фиксировать решение в design.md до правок. Низкий функциональный риск (только оттенок текста).

**Объём: M.** Механика тривиальна после H1; основная работа — согласование канона и триаж скриншот-диффов.

## H3. Шкала fontWeight

**Проблема.** Тема поставляет **YS Text только 400/500/700**. Свип аудита нашёл **13 компонентов** с весами вне шкалы (600/800/900) → браузер синтезирует faux-bold, типографика «плывёт» относительно реального Yandex Pay:
- вес **600**: `yp-badge` (accent), `yp-button` (не-inverted-gray), `yp-cpqr-sheet-frame` / `yp-cpqr-status-bar` (статус-время), `yp-screen` (title), `yp-best-profit-base-card-mini`, `yp-ctyp-payment-page`, `yp-loyalty-badge`;
- вес **800**: `yp-cashback-badge`, `yp-loyalty-badge`;
- вес **900**: `yp-app-home-loans` / `savers` / `vitrina` / `more-important` (h2-заголовки).

Фикс-волна A решает это **округлением к 700** (non-breaking, в скоупе F6). Но округление — потеря дизайн-намерения (600 ≠ 700, 900 — намеренно жирный заголовок витрины).

**Гипотеза.** Развилка (решается дизайном при работе над theme v5):
- **Вариант A — расширить шкалу**: добавить в тему начертания **600 (semibold)** и **900 (black)** реальными файлами YS Text. Сохраняет исходный замысел, faux-bold исчезает по-настоящему.
- **Вариант B — канонизировать 400/500/700**: официально закрепить трёхступенчатую шкалу, компоненты приводятся к ней (то, что и делает фикс-волна округлением). Проще, но h2-витрины теряют «black».

Рекомендация: если файлы 600/900 доступны у Yandex Pay — Вариант A для заголовочных семейств (app-home), 600→канон для UI-текста; иначе Вариант B.

**Ожидаемый эффект.** Отсутствие faux-bold; предсказуемый вес во всех рендерах; шкала темы = закрытый контракт для авторов.

**Миграционный путь.** Вариант A: добавить `@font-face`/токены веса в theme v5 (аддитивно, сосуществует); компоненты, уже округлённые фикс-волной к 700, вернуть к 600/900 там, где так задумано. Вариант B: фикс-волна уже всё сделала — гипотеза сводится к документированию шкалы в design.md. **Важно:** фикс-волна округляет к 700 сейчас; если позже выбран Вариант A, часть округлений придётся откатить — зафиксировать список затронутых компонентов, чтобы откат был точечным.

**Риски.** Вес шрифта — заметное визуальное изменение; вариант A требует наличия файлов начертаний (внешняя зависимость от Yandex Pay). Двойная работа, если фикс-волна округлит, а потом выберут A — минимизируется ранним решением развилки.

**Объём: S** (Вариант B, документирование) — **M** (Вариант A, файлы + токены + откат округлений).

## H4. Миграция numeric spacing → `space()`-токены

**Проблема.** Адопция ABI-spacing минимальна: **`space()`-токены (ABI v3) используют лишь 3 компонента** (`yp-box`, `yp-block`, `yp-app-home-shell`), **72 компонента — ABI v1** без runtime-импорта, вся геометрия хардкодом. Помечено `breaking` в реестре: `yp-text` (ABI v1, `space(props)` кладёт **сырые числа** в margin/gap), `yp-skeleton` (сырые числовые spacing-токены мимо темы), `yp-spacer` (numeric-шкала — заявленный legacy-режим). Перевод шкалы «числа → токены» ломает существующие доки, задающие пиксельные значения.

**Гипотеза.** Мигрировать spacing-пропы приоритетных компонентов с сырых чисел на `space()`-токены темы (ABI v1 → v3). Начать с `yp-text` (23 использования — топ-3 каталога) и `yp-skeleton`; `yp-spacer` — оценить отдельно (у него numeric-scale задекларирован как legacy с линтами, триаж аудита оставил его; возможно, оставить как есть).

**Ожидаемый эффект.** Единая spacing-шкала темы вместо магических чисел; отступы масштабируются темой; согласуется с эталоном `yp-box`. Закрывает класс A ABI-адопции для самых используемых примитивов.

**Миграционный путь.** Breaking, т.к. меняется тип/интерпретация spacing-пропа (число → токен-ключ). Стратегия:
1. **Новая версия компонента** с token-пропами; старая версия остаётся active для запиненных доков (версии immutable — старые пины рендерятся).
2. **Сосуществование**: либо принять оба (число → трактуется как px legacy, токен-ключ → `space()`), либо ввести отдельный проп и депрекейтнуть числовой.
3. **Миграция доков**: скрипт «число → ближайший токен», re-pin по регламенту. Для `yp-text` (23 дока) — прогнать сверку значений против шкалы токенов.

**Риски.** Округление к ближайшему токену может сдвинуть пиксели — триаж диффов. `yp-spacer` legacy-режим триаж уже решил не трогать — не расширять скоуп без нужды. Ломает доки, полагающиеся на точный px.

**Объём: M.** Основная работа — `yp-text` и миграция его 23 доков; `yp-skeleton` дёшево.

## H5. Консолидация near-duplicates

**Проблема.** Аудит зафиксировал кластеры почти-дублей (2 подтверждены verify как `breaking`):
- **`yp-promo-banner` ≈ `yp-banner-mid`** (major, confirmed): идентичный layout (`minWidth:160`, `padding:"20px 0 20px 20px"`, `space-between`), тот же CTA `#2e2f33`/текст `rgba(255,255,255,.98)`, tone-фоны, `borderRadius 20`, событие `press`. Отличия — набор enum и ширины.
- **`yp-app-home-savers` ≈ `yp-app-home-loans`** (major, confirmed): дословно совпадающие section-обёртка (h256, r24, тень `0 4px 10px rgba(0,0,0,.05)`), h2-бейдж-градиент, rail из трёх карточек 172×172. Отличие — data-атрибут и optional caption/image третьей карточки. **Триаж аудита понизил до осознанного fork** — фикс-волна не сводит.
- Смысловые пересечения (не помечены breaking, для design.md): `yp-panel` ↔ `yp-screen` (роли header/content/footer), discount-badge паттерн (`yp-badge` / `yp-discount-info-with-cashback` / `yp-split-discount-info`), статус-бары с часами в разных семействах (`yp-cpqr-sheet-frame` / `yp-cpqr-status-bar` / `yp-app-home-chrome` дублируют реализацию времени), tooltip-поверхность (`yp-tooltip` / `yp-promo-tooltip`).

**Гипотеза.** Развилка по каждому кластеру:
- **Слияние API** (breaking): оставить более конфигурируемый компонент каноническим, второй → deprecated/alias. Меняет id/схемы, ломает доки.
- **Внутренний shared-helper** (non-breaking): вынести общую section-обёртку / BadgeHeading / StatusBar во внутренний хелпер на уровне сборки **без изменения публичных пропов** — устраняет рассинхрон кода, но не сокращает каталог. Для savers/loans аудит прямо предлагает этот путь.

Рекомендация: banner-пара — кандидат на слияние (высокий риск рассинхрона, разные семейства); savers/loans — shared-helper (fork осознанный); статус-бар/tooltip — общий внутренний примитив.

**Ожидаемый эффект.** Меньше рассинхрона при правках (одна правка вместо двух-трёх); меньше «магии» в каталоге; чётко разграниченные роли в design.md.

**Миграционный путь.** Слияние: новый канонический компонент, старый → deprecated (рендерится по пинам), доки мигрируются на канон и re-pin. Shared-helper: чисто внутренняя правка source, non-breaking, проходит как обычный publish (можно даже в фикс-волну, но триаж вынес в бэклог во избежание расширения скоупа). design.md обязан зафиксировать разграничение ролей `panel`/`screen`/`scroll-area`, discount-badge, status-bar — до консолидации.

**Риски.** Слияние ломает доки и требует миграции; неверное разграничение ролей закрепит путаницу — сначала design.md, потом код. Shared-helper безопасен, но не уменьшает число компонентов.

**Объём: M–L** (зависит от числа взятых кластеров и выбора слияние vs helper).

## H6. `enum` для `currency`

**Проблема.** `currency` — **свободная строка** (`z.string().min(1)`) минимум в `yp-discount-info-with-cashback` и `yp-split-discount-info` (помечено `breaking`). Форматтер спецкейсит только `RUB` (`c==="RUB"?"₽":c`), иначе печатает код как есть; свободная строка допускает опечатки, а при undefined даёт «`undefined`» в сумме (отдельные minor-находки по `yp-custom-payment-button`, `yp-badge`). Editor-controls не может дать выбор из списка.

**Гипотеза.** Заменить `currency: z.string()` на `z.enum(["RUB", ...])` (набор реально поддерживаемых валют) во всех компонентах с денежным форматированием. Даёт editor-выпадашку, отсекает опечатки, делает форматтер полным (маппинг символа на каждый enum-член).

**Ожидаемый эффект.** Валидные валюты гарантированы схемой; редактор показывает выбор; исчезает «undefined ₽»/печать сырого кода; форматтер покрывает весь enum.

**Миграционный путь.** Breaking — сужение схемы ломает доки со строковыми значениями вне enum. Стратегия:
1. Инвентарь фактических значений `currency` во всех живых доках (скрипт по 33 докам) — определить реальный набор.
2. Новая версия компонента с `z.enum`; убедиться, что все живые доки попадают в enum (аудит: подавляющее большинство — RUB).
3. Расширять форматтер маппингом символов синхронно с enum.
4. Миграция несоответствующих доков + re-pin.

**Риски.** Док со значением вне enum перестанет валидироваться при импорте/сохранении — обязателен предварительный инвентарь. Низкий, т.к. фактически используется RUB.

**Объём: S.** Малое число компонентов, механическая правка схемы + форматтер.

## H7. Asset-URL контракт для URL-пропов

**Проблема.** URL-пропы задекларированы как свободные строки, минуя пайплайн ассетов. Эталон рядом: `yp-app-home-chrome` применяет строгий `z.string().regex(/^\/api\/assets\/asset_[a-f0-9]{64}$/)`, а соседний **`yp-app-home-vitrina` — `z.string().min(1).max(500)`** для всех `*ImageUrl` (помечено `breaking`), хотя example ссылается на `/api/assets/...`. Схема допускает произвольные/внешние URL. Аналогично свободные url-пропы в `yp-app-home-vitrina`, `yp-cpqr-*`, `yp-custom-*`. Отдельно — 3 компонента (`yp-random-avatar`, `yp-maps-review-banner`, `yp-ctyp-payment-page`) с base64 data-URI в исходнике (~253KB); их вынос в ассеты — **non-breaking, в фикс-волне (F2)**, но он не закрывает контракт схемы.

**Гипотеза.** Сузить URL-пропы, ссылающиеся на изображения, до **asset-url регекспа** (как в `yp-app-home-chrome`). Внешние/произвольные URL запрещаются схемой — всё через `POST /api/assets` (пиннинг, дедуп, кэш).

**Ожидаемый эффект.** Все изображения проходят пайплайн ассетов (дедуп по sha256, кэширование, отсутствие «мёртвых» внешних ссылок); editor-валидация ловит невалидный ref; согласованность с `chrome`-эталоном.

**Миграционный путь.** Breaking — сужение схемы. Стратегия:
1. Инвентарь значений url-пропов в живых доках; залить внешние/inline-изображения в ассеты (F2 фикс-волны уже делает это для 3 data-URI компонентов).
2. Новая версия компонента с regex-схемой; проверить, что все живые доки соответствуют.
3. Миграция доков на asset-ref + re-pin.
4. Альтернатива (если нет ресурса на миграцию): оставить свободную строку и полагаться на дисциплину авторов + скилл `yandex-pay` — но это не закрывает риск.

**Риски.** Док с внешним URL сломается на валидации — обязателен инвентарь и предзаливка ассетов. Средний: url-пропов много, но большинство уже указывают на `/api/assets/...`.

**Объём: S–M.** Регексп тривиален; работа — инвентарь + предзаливка + миграция доков.

## H8. Единое представление знака Плюса

**Проблема.** Знак Yandex Plus рендерится **тремя разными способами**: registry-ассет (`yp-plus-badge`, `yp-snippet-plus` — эталон), кириллическая **«Я»** глифом (`yp-cashback-badge` цветом `#7b42f6`, `yp-loyalty-badge` `#7b42f6`, и родственные), символ **«✦»** (`yp-plus-return`, вес 700). Плюс расходящиеся градиенты плюс-глифа (`plus-glyph-gradient` 90deg в `yp-plus-progress` vs 135deg в батче; `#edeff2` vs `#f2f3f5`). Completeness-critic отдельно отметил «Я вместо фирменного знака». Фикс-волна **осознанно не правит** (F5: визуальное изменение при явных пропах — продуктовый вопрос).

**Гипотеза.** Ввести **единый registry-ассет знака Плюса** (SVG/webp фирменного знака) и перевести все Plus-компоненты на него, убрав глифы «Я»/«✦». Градиенты Плюса свести к канону (135deg, единые стопы) как токен темы (пересекается с H1).

**Ожидаемый эффект.** Фирменный знак Плюса вместо суррогатных глифов → визуальный паритет с реальным Yandex Pay; один источник знака (правка в одном ассете); согласованный градиент.

**Миграционный путь.** Breaking (визуальное изменение). Стратегия:
1. Залить канонический ассет знака Плюса (`POST /api/assets`).
2. Обновить компоненты: заменить глиф-рендер на `<img>`/inline-ассет; градиент → токен (после H1).
3. Это меняет пиксели при уже заданных пропах → новая версия компонента, прод-baseline + триаж диффов (ожидаемое улучшение).
4. design.md фиксирует «знак Плюса = registry-ассет `asset_<sha>`, глифы запрещены» как контракт.

**Риски.** Заметное визуальное изменение на многих экранах — нужен baseline и явный триаж; зависимость от наличия качественного ассета знака. Средний.

**Объём: M.** Ассет + правка ~6–8 Plus-компонентов + триаж диффов; градиент-часть дешевле после H1.

## H9. Детерминизм `yp-random-avatar` (seed-проп)

**Проблема.** `yp-random-avatar` использует **`Math.random`** в `useMemo` → рендер **недетерминирован**: ломает визуальные снапшоты (фикс-волна вынуждена исключить компонент из pixelmatch-гейта, оставив только ручной осмотр — риск в плане W1). Тот же компонент несёт 22 инлайн-base64 webp (~197KB) — вынос в ассеты идёт в фикс-волне (F2), но недетерминизм ею не лечится.

**Гипотеза.** Добавить опциональный **`seed`-проп** (число/строка); при заданном seed выбор аватара детерминирован (seeded PRNG вместо `Math.random`). Без seed — сохранить текущее случайное поведение (обратная совместимость).

**Ожидаемый эффект.** Воспроизводимый рендер в доках и снапшотах → компонент возвращается в pixelmatch-гейт; авторы могут зафиксировать конкретный аватар; текущее поведение без seed не меняется.

**Миграционный путь.** Breaking-lite — только **добавление** optional-пропа (расширение схемы). По конвенции фикс-волны схемы не расширяются (F1), поэтому это бэклог, но риск минимален:
1. Добавить `seed?: z.union([z.string(), z.number()]).optional()` + seeded-PRNG; при `undefined` — `Math.random` как сейчас.
2. Доки не мигрируются принудительно; желающие детерминизма проставляют seed.
3. Новая версия компонента; старые пины рендерятся по-старому.

**Риски.** Минимальные — аддитивное расширение схемы, дефолт сохраняет поведение. Технически «не non-breaking» лишь по букве конвенции (расширение схемы), функционально безопасно.

**Объём: S.** Один компонент, seeded-PRNG (например, mulberry32 от хеша seed).

## H10. Runtime-адопция (v1 → v2/v3) как стратегический долг

**Проблема.** **72/100 компонентов — ABI v1** без runtime-импорта (палитра/геометрия хардкодом); ABI v2 (`token()`/`Icon`) — 25, ABI v3 (`space()`) — **всего 3**. H1–H4 закрывают палитру/spacing точечно, но системный долг — массовая непринятость runtime — остаётся.

**Гипотеза.** Отдельная стратегическая инициатива: помиграционно поднимать компоненты на актуальный ABI (runtime v2 → v3 → v4 после H1), делая `color()`/`space()`/`token()`/`Icon` дефолтом. Не самостоятельная фича, а зонтик над H1/H2/H4 + постепенное подтягивание оставшихся.

**Ожидаемый эффект.** Каталог становится тематизируемым целиком; новые темы (не только yandex-pay) применяются без правки компонентов; эталон `yp-box` — норма, а не исключение.

**Миграционный путь.** Инкрементально, семействами, поверх H1 (нужны цветовые токены). Каждый шаг — новая версия компонента, fallback = текущий литерал, дифф 0. Приоритет — по частоте использования (`yp-box` 61, `yp-icon` 43, `yp-text` 23, `yp-payment-method-card` 19).

**Риски.** Большой объём; ценность растёт только при появлении второй темы или сильной ребрендинг-задачи — иначе низкий ROI. Держать как фоновый долг, не форсировать.

**Объём: L** (растянутый, фоновый).
